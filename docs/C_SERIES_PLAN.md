# Рекомендованный порядок реализации C-серии и требования уровня premium-продукта

## 1. Общая оценка текущего backlog

В backlog остались задачи C-серии, которые относятся к категории `nice-to-have`, то есть не являются критическими блокерами релиза, но существенно влияют на ощущение качества, удобства и «дороговизны» продукта.

Список оставшихся задач:

* C1 — FTS5-поиск.
* C2 — облачная синхронизация между устройствами.
* C3 — A/B framework.
* C4 — offline export в формате `.apkg` без AnkiConnect.
* C5 — RTL polish.
* C6 — keyboard shortcuts.
* C7 — markdown в заметках.

Рекомендуется реализовывать эти задачи не в порядке номеров, а по влиянию на пользовательский опыт, рискам и стоимости реализации.

# 2. Рекомендованный порядок реализации C-серии

## Tier 1 — реализовать в первую очередь

Эти задачи дают максимальный эффект при умеренной стоимости реализации.

## 2.1. C5 — RTL polish

Так как приложение ориентировано на работу с ивритом, поддержка RTL должна быть не просто рабочей, а безупречной.

Что нужно сделать:

* Провести аудит всех экранов, модальных окон, таблиц, карточек и панелей.
* Проверить правильность направления текста справа налево.
* Проверить отступы, выравнивания, flex/grid-раскладки.
* Проверить иконки, зависящие от направления: стрелки, swipe, назад/вперёд.
* Проверить смешанные строки, где одновременно есть иврит, русский и английский.
* Исправить случаи, где интерфейс визуально «ломается» при RTL.

Оценка: 1–2 дня.

Ожидаемый эффект:

* Приложение будет восприниматься как профессиональный Hebrew-first продукт.
* Уйдут визуальные ошибки, которые сразу создают ощущение сырого интерфейса.
* Повысится доверие к продукту у пользователей, работающих с ивритом.

---

## 2.2. C1 — FTS5-поиск

Поиск — это ежедневная функция. Сейчас поиск реализован через `LIKE`/substring-подход, который может тормозить на больших библиотеках.

Что нужно сделать:

* Добавить FTS5 virtual table в SQLite.
* Настроить синхронизацию через триггеры.
* Поддержать поиск по текстам, предложениям, заметкам и метаданным.
* Учесть иврит, русский и английский.
* Сделать корректное отображение результатов поиска.
* Добавить zero-result suggestions, если ничего не найдено.

Оценка: 2–3 дня.

Ожидаемый эффект:

* Поиск станет быстрее и удобнее.
* Приложение лучше масштабируется на большие библиотеки.
* Появится база для будущего smart-discovery: умных рекомендаций, фильтров, поиска по смысловым группам.

---

## 2.3. C7 — markdown в заметках

Markdown в заметках — небольшая, но заметная premium-функция. В интерфейсе уже есть намёки на markdown-форматирование, например `**жирный текст**` или `==выделение==`.

Что нужно сделать:

* Добавить markdown-preview.
* Поддержать базовые конструкции:

  * жирный текст;
  * курсив;
  * выделение;
  * списки;
  * ссылки;
  * переносы строк.
* Добавить переключатель Edit / Preview.
* Обеспечить безопасный рендеринг без XSS.
* Сохранить текущую простоту заметок.

Оценка: 0.5–1 день.

Ожидаемый эффект:

* Заметки станут удобнее для учебной работы.
* Пользователь сможет структурировать свои мысли, пометки и объяснения.

# 4. Tier 3 — отложить

## 4.1. C2 — cloud sync между устройствами

Облачная синхронизация — потенциально самая ценная функция, но она самая сложная и рискованная.

Почему лучше отложить:

* Требуется авторизация.
* Нужна серверная часть.
* Нужно продумать conflict resolution.
* Нужно решить вопрос приватности.
* Это может противоречить философии `data on device`, если не будет end-to-end encryption.
* Реализация может занять 2–4 недели.

Рекомендуемый подход:

* Не делать полноценную multi-master sync прямо сейчас.
* Начать с optional encrypted backup.
* Рассмотреть синхронизацию через Dropbox, Google Drive или другой user-owned storage.
* Сделать zero-trust подход: сервер не должен иметь доступа к содержимому данных.
* Для быстрого обмена между устройствами можно добавить QR-handoff или ZIP-handoff.

Ожидаемый эффект при реализации:

* Пользователь сможет переносить состояние между устройствами.
* Но стоимость и архитектурные риски слишком высоки для текущего этапа.

---

## 4.2. C3 — A/B framework

A/B framework сейчас преждевременен.

Почему лучше отложить:

* Нужен достаточный пользовательский трафик.
* Без статистически значимой выборки A/B framework не даст пользы.
* Есть риск получить мёртвый код.
* Приоритет лучше отдать качеству базового UX.

Когда делать:

* Когда будет стабильная пользовательская база.
* Когда появятся конкретные гипотезы для проверки.
* Когда будет не менее 1000+ активных пользователей или достаточный объём событий.

Что нужно реализовать:

* Optimistic UI для всех мутаций:

  * сохранение;
  * удаление;
  * редактирование;
  * добавление заметок;
  * изменение тегов;
  * изменение статусов.
* Rollback при ошибке через undo-toast.
* Skeleton-загрузки вместо обычных spinner'ов.
* Реакция интерфейса на действия пользователя менее чем за 100 мс.
* Индикатор сохранения в header активной карточки:

  * `✓ Сохранено 2 секунды назад`;
  * `Сохраняется…`;
  * `Ошибка сохранения — повторить`.

Как это отразится:

* Исчезнет ощущение «зависания».

## 6.2. Ивритская типографика как first-class feature

Что нужно реализовать:

* Подобрать качественный шрифт для иврита.
* Проверить корректный рендеринг огласовок.
* Создать тестовую страницу `/typo` с edge-case'ами:

  * шева;
  * дагеш;
  * патах;
  * камац;
  * шурук;
  * смешанные строки Hebrew + Russian + English.
* Проверить line-height, letter-spacing и vertical alignment.
* Использовать bidi-isolation для смешанных строк.

Как это отразится:

* Тексты на иврите будут выглядеть аккуратно и профессионально.
* Огласовки не будут визуально «прыгать».
* Пользователь будет видеть, что приложение сделано именно под иврит, а не просто адаптировано поверх обычного LTR-интерфейса.

## 6.3. Error gentleness

Что нужно реализовать:

* Убрать `alert()` из пользовательского интерфейса.
* Все ошибки показывать контекстно:

  * рядом с кнопкой;
  * внутри карточки;
  * через toast;
  * через аккуратную inline-плашку.
* Для transient-fail добавить кнопку `Попробовать снова`.
* Для сетевых вызовов добавить retry с backoff:

  * TTS;
  * transliteration;
  * niqqud;
  * upload;
  * export.
* Разделить ошибки на:

  * временные;
  * пользовательские;
  * системные;
  * критические.

Как это отразится:

* Приложение будет меньше раздражать пользователя.
* Ошибки будут восприниматься как контролируемые ситуации, а не как поломка.
* UX станет мягче и профессиональнее.

Что нужно реализовать:

  * экспортировать нечего.

  * `Создать первый текст`;
  * `Добавить заметку`;
  * `Сбросить фильтры`;
  * `Импортировать текст`.
* Для zero-result поиска показывать suggestions.

Как это отразится:

* Пользователь не будет попадать в «пустой экран».
* Каждый пустой сценарий будет подсказывать следующий шаг.
* Продукт станет выглядеть более заботливо и дорого.

# 10. Что будет сделано дальше и как это отразится

## Что будет сделано дальше

Следующим логичным шагом является не начинать с самых крупных и рискованных задач, а сначала закрыть те элементы, которые сильнее всего влияют на ежедневное ощущение качества продукта.

Рекомендуемый порядок:

1. Сначала довести RTL до безупречного состояния.
2. Затем улучшить визуальную зрелость интерфейса: empty states, сохранение, мягкие ошибки, типографика.
3. После этого внедрить быстрый FTS5-поиск.
4. Затем добавить markdown-preview в заметках.
5. После этого перейти к keyboard shortcuts и `.apkg` offline export.
6. Cloud sync и A/B framework оставить на более поздний этап.

---

## Как это отразится на продукте

После первого этапа приложение будет выглядеть заметно более профессионально:

* интерфейс станет аккуратнее;
* иврит будет отображаться корректнее;
* пользователь будет лучше понимать состояние сохранения;
* исчезнут грубые пустые экраны;
* ошибки станут мягкими и понятными;
* появится ощущение быстрого отклика.

После второго этапа продукт станет удобнее в ежедневном использовании:

* поиск станет быстрым;
* заметки станут более полезными;
* появится базовая аналитика прогресса;
* пользователь начнёт видеть результат своих занятий.

После третьего этапа продукт станет ближе к premium-инструменту:

* появятся горячие клавиши;
* экспорт в Anki станет независимым от AnkiConnect;
* приложение будет лучше работать в offline-сценариях;
* появится больше уверенности, что продукт можно использовать регулярно и на разных устройствах.
