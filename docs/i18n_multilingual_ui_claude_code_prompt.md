# Задача для Claude Code: мультиязычный интерфейс RU / EN / HE для профессионального premium-приложения

## Контекст проекта

Мы разрабатываем профессиональное premium-приложение в репозитории `tts-prototype-android`.

Текущий интерфейс в основном русскоязычный, но в проекте могут оставаться англоязычные пользовательские артефакты: кнопки, подписи, статусы, сообщения ошибок, tooltips, aria-label, placeholder, заголовки, элементы меню, диагностические строки, модальные окна, toast-сообщения, badge/status labels и другие видимые пользователю строки.

Необходимо разработать и реализовать полноценную архитектуру мультиязычного интерфейса:

- Русский — текущий основной язык и язык по умолчанию.
- Английский — дополнительный полноценный язык интерфейса.
- Иврит — дополнительный полноценный язык интерфейса с корректной RTL-поддержкой.

Важно: это не одноразовая замена строк, а системная premium-реализация i18n, пригодная для дальнейшего масштабирования приложения.

---

## Главная цель итерации

Разработать и реализовать требования для мультиязычного интерфейса приложения:

1. Провести аудит текущего интерфейса и найти все пользовательские строки.

3. Реализовать английскую и ивритскую локализации.
4. Добавить premium-механизм переключения языка в интерфейсе.
5. Обеспечить корректную поддержку RTL для иврита.
6. Минимизировать риск регрессий в Classic Mode, Library, TTS, таблицах, мобильном интерфейсе и существующих сценариях.
7. Обновить документацию и добавить evidence/DoD по результатам реализации.

## Архитектурные требования к i18n

### 1. Нельзя продолжать hardcoded UI strings

Нужно минимизировать или устранить прямые пользовательские строки в HTML/JS.

Необходимо внедрить централизованный слой локализации.

Предпочтительный подход:

- создать i18n-модуль, например:
  - `public/i18n/index.js`
  - `public/i18n/locales/ru.js`
  - `public/i18n/locales/en.js`
  - `public/i18n/locales/he.js`

или другой подход, если он лучше соответствует текущей архитектуре проекта.

Требования:

- все новые пользовательские строки должны идти через `t(key)`;
- должен быть fallback на русский язык;
- при отсутствии ключа должна быть безопасная деградация;
- отсутствующие ключи должны логироваться в dev/debug-режиме, но не ломать UI;
- нельзя ломать существующие глобальные функции и id-контракты в `public/index.html`.

---

### 2. Русский язык — default locale

По умолчанию интерфейс должен открываться на русском языке.

При первом запуске:

- если пользователь уже выбирал язык — использовать сохранённый выбор;
- если нет — использовать русский;
- опционально можно учитывать `navigator.language`, но только если это не создаёт неожиданного поведения;
- при сомнении default должен быть `ru`.

---

### 3. Переключатель языка

Нужно реализовать premium-решение для переключения языка.

Допустимые варианты:

- компактный selector в верхней панели;
- пункт в настройках;
- language button/dropdown;
- комбинированное решение: быстрый переключатель + настройка в панели.

Требования к UX:

- пользователь должен явно видеть текущий язык;
- названия языков должны быть нативными:
  - `Русский`
  - `English`
  - `עברית`
- не использовать флаги как основной способ выбора языка, потому что язык не равен стране;
- переключение должно происходить без перезагрузки страницы, если это технически разумно;
- выбор должен сохраняться между сессиями;
- переключатель должен корректно работать на desktop и mobile;
- переключатель не должен ломать существующую адаптивную вёрстку Classic Mode.

---

### 4. Поддержка иврита и RTL

Для иврита необходимо реализовать полноценную RTL-поддержку интерфейса.

При выборе `he`:

- установить `document.documentElement.lang = "he"`;
- установить `document.documentElement.dir = "rtl"`;
- применить RTL-aware layout там, где это необходимо;
- убедиться, что основные блоки интерфейса не ломаются;
- текстовые поля с ивритским вводом должны оставаться удобными;
- таблицы с колонками Hebrew / Niqqud / Transliteration / Russian Translation не должны стать визуально хаотичными;
- direction для данных внутри таблицы должен быть осмысленным:
  - ивритские колонки — RTL;
  - русские и английские переводы — LTR или auto, если так лучше;
  - SBL transliteration — LTR;
- кнопки, панель действий, статусы и меню должны быть читаемыми в RTL.

Важно: RTL не должен быть просто `dir="rtl"` на весь документ без проверки layout. Нужно проверить реальные экраны.

---

Переводить нужно все user-facing элементы, включая:

- заголовки;
- кнопки;
- меню;
- вкладки;
- placeholder;
- tooltip;
- aria-label;
- toast/status messages;
- confirmation dialogs;
- error dialogs;
- empty states;
- badges;

- настройки;
- mobile-specific labels;
- Library UI;
- Classic Mode UI;
- TTS UI;
- table edit mode UI;
- import/export UI;
- provider status UI, где это уместно.

---

### 6. Что НЕ переводить или переводить осторожно

Не нужно переводить технические идентификаторы, если они являются частью диагностики или API-контракта:

- provider ids:
  - `google`
  - `gemini`
  - `madlad`
  - `local_neural_tts_piper`
  - `hebrew_local_piper`
- runtime ids;
- backend ids;
- model ids;
- API endpoint paths;
- log-only developer messages;
- internal function names;
- localStorage keys;
- CSS class names;
- DOM ids;
- тестовые fixture ids;
- machine-readable provenance fields.

Но если эти значения показываются пользователю, рядом должна быть понятная локализованная подпись.

Пример:

- плохо: `Provider: local_neural_tts_piper`
- лучше:
  - RU: `Провайдер: локальный Hebrew Piper (local_neural_tts_piper)`
  - EN: `Provider: Local Hebrew Piper (local_neural_tts_piper)`
  - HE: `ספק: Hebrew Piper מקומי (local_neural_tts_piper)`

## Требования к качеству переводов

Переводы должны быть не машинально-дословными, а UI-естественными.

Русский:

- профессиональный, понятный, без англицизмов там, где есть нормальный русский термин;
- сохранить текущий стиль приложения;
- исправить оставшиеся английские UI-артефакты, если они не являются техническими идентификаторами.

Английский:

- clean professional UI English;
- не перегружать длинными фразами;
- использовать стандартные UI-термины.

Иврит:

- естественный современный интерфейсный иврит;
- не использовать слишком буквальные кальки с русского;
- учитывать RTL;
- для сложных технических терминов допустимо сохранять английский technical id в скобках.

## Требования к DOM binding

Нужно выбрать безопасный способ применения переводов к существующему UI.

Допустимые варианты:

1. `data-i18n` атрибуты:
   - `data-i18n="classic.generateTable"`
   - `data-i18n-placeholder="classic.inputPlaceholder"`
   - `data-i18n-title="classic.tooltip"`

2. JS binding через helper:
   - `setText(el, t("..."))`
   - `setPlaceholder(el, t("..."))`

3. Гибридный подход.

Важно:

- не ломать существующие id;
- не ломать event listeners;
- не ломать inline handlers, если они используются;
- не ломать текущую инициализацию `public/index.html`;
- обеспечить повторное применение переводов после динамического рендера блоков;
- предусмотреть перевод динамически создаваемых элементов.

Требования:

- поддерживать только разрешённые значения:
  - `ru`
  - `en`
  - `he`
- если значение неизвестное — fallback на `ru`;
- не ломать старые localStorage values.

## UI/UX CONSTRAINTS

Обязательно соблюдать следующие UI/UX-ограничения:

1. Не использовать фиксированные высоты для длинного контента без scroll/overflow-стратегии.
2. Все длинные панели, таблицы, списки и модальные окна должны быть прокручиваемыми.
3. На мобильном интерфейсе не должно быть горизонтального overflow всей страницы.
4. Таблица может иметь локальный горизонтальный скролл, но не должна ломать viewport.
5. Учитывать safe-area/insets для мобильных браузеров.
6. Кнопки должны оставаться доступными для пальца на мобильном экране.
7. Не прятать критичные действия вне видимой области без понятной навигации.
8. Поддержать keyboard accessibility:
   - focus states;
   - tab order;
   - Enter/Escape где уместно;
   - aria-label для language selector.
9. Не ломать screen-reader semantics.
10. Проверить mobile portrait layout.
11. Проверить desktop layout.
12. Проверить RTL layout отдельно.
13. Добавить evidence в документацию: какие экраны проверены и что именно не сломано.

## Требования к тестированию

Нужно добавить или обновить тесты в соответствии с текущей тестовой инфраструктурой проекта.

Минимальный набор проверок:

1. Locale registry содержит `ru`, `en`, `he`.
2. Для каждого ключа из `ru` есть соответствующий ключ в `en` и `he`.
3. `t(key)` возвращает корректный fallback.
4. Unknown key не ломает UI.
5. Language selector сохраняет выбор.
6. При выборе `he` устанавливаются:
   - `lang="he"`
   - `dir="rtl"`
7. При выборе `ru` и `en` устанавливается:
   - `dir="ltr"`
8. Критичные кнопки Classic Mode получают переведённый текст.
9. Dynamic status messages используют i18n.
10. Нет очевидных оставшихся hardcoded English user-facing строк в основных UI-зонах, кроме разрешённых technical ids.

Если в проекте нет подходящей тестовой инфраструктуры для DOM/i18n, добавить lightweight smoke-тесты или scripts, не создавая тяжёлой зависимости без необходимости.

## Definition of Done

Итерация считается завершённой только если выполнено всё ниже:

1. Интерфейс поддерживает русский, английский и иврит.
2. Русский язык остаётся языком по умолчанию.
3. Language selector виден, понятен, доступен и работает на desktop/mobile.
4. Выбор языка сохраняется между сессиями.
5. При выборе иврита включается RTL.
6. При возврате на русский/английский включается LTR.
7. Основные пользовательские строки вынесены в i18n.
8. Неуместные англоязычные артефакты в русском интерфейсе заменены на русский.
9. Технические provider ids и machine-readable поля не сломаны.
10. Classic Mode продолжает работать.
11. Library продолжает работать.
12. TTS UI продолжает работать.
13. Mobile layout не деградировал.
14. RTL layout проверен отдельно.
15. Добавлены тесты или smoke scripts.
16. Обновлена документация.
17. В `docs/I18N_MULTILINGUAL_UI_DOD.md` есть evidence: что проверено, какими командами, какие результаты.
18. `git diff` понятен и не содержит случайных массовых переименований технических id.
19. Нет регрессии существующих тестов.
20. Нет silent failure при отсутствующем ключе локализации.

## Acceptance checklist для ручной проверки

После реализации вручную проверить:

### RU

- открыть приложение;
- убедиться, что интерфейс по умолчанию на русском;
- проверить Classic Mode;
- проверить Library;
- проверить TTS panel;
- проверить table edit mode;
- убедиться, что не осталось неуместных английских UI-слов.

### EN

- переключить язык на English;
- проверить, что все основные кнопки, заголовки, статусы и ошибки на английском;
- перезагрузить страницу;
- убедиться, что English сохранился.

### HE

- переключить язык на עברית;
- проверить, что интерфейс стал RTL;
- проверить, что header, panels, buttons, Library и Classic Mode читаемы;
- проверить таблицу:
  - Hebrew/Niqqud RTL;
  - Transliteration LTR;
  - Russian/English translation не ломается;
- перезагрузить страницу;
- убедиться, что Hebrew сохранился.

### Regression

- сгенерировать таблицу из тестового ивритского текста;
- проверить niqqud;
- проверить transliteration;
- проверить translation;
- включить edit mode;
- изменить ячейку;
- сохранить/закрыть edit mode;
- открыть Library;
- выполнить поиск;
- применить сортировку;
- вернуться назад.

## Ограничения

Не делать:

- не переписывать весь `public/index.html` без необходимости;
- не ломать существующие DOM ids;
- не переименовывать provider ids;
- не менять API-контракты без явной необходимости;
- не добавлять тяжёлые i18n-фреймворки без доказанной пользы;
- не делать частичный перевод только главного экрана;
- не оставлять иврит без RTL;
- не смешивать машинные technical ids и user-facing labels;
- не скрывать ошибки отсутствующих переводов.

## Финальный отчёт

После реализации предоставить итоговый отчёт:

1. Что было найдено при аудите.
2. Какая i18n-архитектура реализована.
3. Какие файлы изменены.
4. Какие языки поддержаны.
5. Как работает переключение языка.
6. Как реализован RTL.
7. Какие строки намеренно не переводились и почему.
8. Какие тесты выполнены.
9. Какие manual smoke checks выполнены.
10. Known limitations.
11. Commit-ready summary.

Отчёт должен быть конкретным, с указанием файлов и результатов проверок.
