## §0. TL;DR

Продукт начинает **сам владеть петлёй повторения**. Три связанные части = одна программа:

1. **#5 Канон-модель памяти:** истина о памяти слова = **append-only лог ревью** (`review_log`, новая OPFS-таблица), ключ = единый канон-кейер (один экспортируемый модуль для ВСЕХ пишущих путей + конформанс-гейт). Состояние планировщика (stability/difficulty/due) — **производный кэш**, пересчитываемый из лога (R9; сид легаси-состояния — тоже строка лога, поэтому independent-oracle гейт замкнут с первого дня). Ручной статус `l1..l4/known/ignore` — отдельная асертивная ось.
2. **#2 FSRS:** один pure-движок `public/js/fsrs-core.js` (UMD, детерминизм, nowMs-инъекция, golden-векторы с запиненным поколением/весами) заменяет ОБА планировщика — SM2-lite Зала (`nextSrs`) и SM2-стаб Студии (`computeSM2`). **Lazy-seed:** ни одно существующее due не передвигается миграцией; контракт «Again → due сейчас» сохраняет же-сессионный ре-тест провалов; перед переключением — офлайн shadow-diff на реальном профиле.
3. **#6 Reading-native retrieval (моат):** due-слова видны в живом чтении (тихий маркер, только exact-уверенные) + **reveal-then-grade** карточка на тап (попытка про себя → показ ответа → «Знал/Не знал»). Пассивная встреча ≠ ревью; самоотчёт из чтения НЕ двигает ручной уровень (развилка D8). Ни у LingQ, ни у Clozemaster этого нет.

**Приоритет поверхностей: Зал первый.** Петля Зала — живая и активно используемая владельцем прямо сейчас; Студия-Trainer — стаб (подтверждено разведкой: две системы не пересекаются ни одной записью). Anki эволюционирует из «слоя ревью» в **опционального компаньона**: экспорт и локальный read-back остаются, конфликт решается слиянием логов + пересчётом, не перезаписью состояний.

**Попутно закрываются найденные разведкой/критикой реальные дыры сегодняшнего кода:** `word_status` и `study_day` не экспортируются в бандл (перенос устройства теряет ручную ось и стрик); importBundle Pass 8/12 дублирует `srs_review_events` и разрушает `anki:<id>`-идентичность `events`; мёртвая ссылка `DELETE FROM srs_reviews`; фиктивная задвоенная нумерация миграций («061»×2 при реальной длине массива 40).

### 2.6 Пробелы, которые программа закрывает
- Нет истории ревью Зала → FSRS-калибровка, оптимизатор, честная метрика удержания и R9-пересчёт состояния невозможны.

- Бандл-перенос теряет память (2.5).

## §3. Дизайн #5 — канон-модель памяти

### 3.1 Три оси (R9)
| Ось | Что это | Где живёт | Кто пишет |
|---|---|---|---|
| **Асертивная** | «я утверждаю, что знаю на уровне X» | `word_status.status` (l1..l4/known/ignore/new) | пользователь (тап/long-press/лич-игнор) + `nextLevel` по ТЕСТИРУЕМОМУ recall (связка); самоотчётные каналы — НЕТ (D8) |
| **Событийная (ИСТИНА)** | что реально происходило: попытки припоминания + seed-снимки | **НОВОЕ: `review_log`** (append-only) | recall Зала, Trainer Студии, Anki-импорт, reading-тап (P5), seed-события (P2/P3) |
| **Производная (КЭШ)** | состояние планировщика: S/D/due | `word_status.srs_*` (+новые колонки) и `srs_cards.meta_json.fsrs` | ТОЛЬКО fsrs-core фолдом лога; пересчитываемо из лога |

### 3.2 `review_log` (мигр. 041 — РЕАЛЬНЫЙ индекс, см. §3.6) — каноническая таблица
```sql
CREATE TABLE review_log (
  id          TEXT PRIMARY KEY,   -- контент-детерминированный, глобально уникальный (ниже)
  item_key    TEXT NOT NULL,      -- канон-кейер §3.3 | 'sent:<text_key>#<order_index>#<template>' для sentence-карт
  kind        TEXT NOT NULL DEFAULT 'review',  -- 'review' | 'skip' | 'seed'
  reviewed_at TEXT NOT NULL,      -- ISO ms; при инжесте future-ts клампится (meta.ts_clamped=1)
  grade       INTEGER,            -- 1..4 (Again/Hard/Good/Easy); бинарная петля Зала пишет 1|3; NULL для kind='seed'
  source      TEXT NOT NULL,      -- 'room-recall' | 'room-due-queue' | 'reading-tap' | 'studio-trainer' | 'anki' | 'seed-sm2'
  channel     TEXT,               -- read|listen|reverse|dictate × mc|tiles|typed; NULL для anki/seed
  latency_ms  INTEGER,
  meta_json   TEXT NOT NULL DEFAULT '{}'
);
CREATE INDEX ix_review_log_item ON review_log(item_key, reviewed_at);
CREATE INDEX ix_review_log_time ON review_log(reviewed_at);
```
**id — глобальная уникальность + идемпотентность (BLOCKER-фиксы B5):**
- app-события: `app:<sha1(item_key|reviewed_at|grade|source|channel)[:20]>` — контент-детерминирован → merge двух устройств не теряет чужие события и не дублирует свои; монотонные per-device счётчики ОТВЕРГНУТЫ.
- Anki: `anki:<anki_review_id>` — id берётся из **payload/`anki_review_id`**, НИКОГДА из `events.id` (Pass 12 его пере-генерит).
- Seed: `seed:<item_key>` (один сид на item; повторный сид невозможен по PK).
- Бэкфилл Студии (если пройдёт замер, §8 P·0): `sre:<sha1(card_id|reviewed_at|rating)[:20]>` — детерминирован от контента, а не от нестабильного event-uuid.

**meta_json — обязательный минимум провенанса (B2, R9/R11-фиксы):**
- улики идентичности: `{surface, niqqud, pos, pid, lemma, keyer_version}` — сырьё для будущего ре-кеинга;
- предсказатель на момент записи: `{scheduler:{scheme, engine_version, request_retention}}`;
- флаги честности: `skipped` (наследие kind), `postTeach:1` (немедленный пост-teach тест — исключается из фита весов), `ts_clamped`, `ts_fabricated`;
- anki: `{anki_card_id, ord, anki_type}` (направление карты — провенанс не теряем);
- reading-tap: `{text_key, confidence}`;
- seed: `{interval, reps, lapses, scheme:'sm2-lite', seedAlgoVersion}` — снимок легаси-состояния;
- бэкфилл: `{keyed_from:{note_id, body_hash, keyed_at}}` — ключ считался от СЕГОДНЯШНЕГО body, это записано честно.

**Seed-события (BLOCKER-фикс B4, сошлись R9+R11+R10):** сид легаси-состояния — НЕ фабрикация истории, а честная asserted-запись факта «состояние сидировано из SM2-снимка в момент T» (kind='seed', grade NULL). `replay()` стартует item от seed-строки (его watermark); строки раньше watermark в state-фолд не входят (остаются для метрик). Это замыкает лог: состояние ЛЮБОГО слова выводимо из лога → independent-oracle гейт имеет зубы с первого дня, включая окно P0→P2 (ревью, записанные в лог до сида, поглощаются watermark'ом — двойного применения нет).

**Пишущие пути (P0):** `checkTrainAnswer` (Зал; skip → kind='skip', grade 1) · `srs.reviewCard` (Студия) · Anki-инжест (§5). **Контракт фолда:** live-путь и `replay()` применяют строку ОДНОЙ И ТОЙ ЖЕ функцией fsrs-core (skip-строки — ровно как live; расхождение исключено по построению).

**Бандл (BLOCKER-фикс B6):** P0 добавляет в `exportBundle`/`importBundle` ТРИ таблицы: `review_log` (merge по PK, INSERT OR IGNORE — id глобально уникальны), `word_status` (merge: newer-`updated_at`-wins, COALESCE-сохранение source-якорей — паттерн 060), `study_day` (merge: per-day MAX(recalls, available)). Заодно чинятся Pass 8 (переносить исходные PK `srs_review_events` — они уникальны; не uuid()) и Pass 12 (сохранять `anki:<id>`-события с исходным id). Раунд-трип всех трёх + merge-фикстура «два устройства, пересекающиеся истории» — в `smoke:memory-canon`.

**Мульти-таб:** новые localDb-функции идут через существующий generic-прокси; в gate-consumers-sweep включить проверку follower-вкладки.

### 3.3 Ключ-канон (BLOCKER-фиксы B1+B2+B3)
- **Один физический кейер:** новый крошечный UMD **`public/js/lemma-canon.js`** (pure, без DOM) = каноническая функция ключа (`pid:<id>` | `<strip(лемма|слово)>#<pos>` + огласовко-производный surface + function-link-pid хук) с `KEYER_VERSION`. НА НЕГО переводятся: `statusKeyForCard` (reader-morph), `_noteLemmaKey` (local-db — сейчас БЕЗ никуд-стрипа: чинится), Anki-инжест-маппинг, бэкфилл. `PealimFunctionLinks`-карта — data-файл, подгружается и в Студии (иначе функц-слова ключуются иначе). **Конформанс-гейт** в `smoke:memory-canon`: одна фикстура body/span → байт-идентичный ключ со ВСЕХ пишущих путей.
- **Стратегия ре-кеинга (ключ нестабилен во времени — резолвер/датасет улучшаются):** (а) каждая строка лога несёт сырые улики идентичности в meta (выше); (б) операция re-key = батч-миграция `item_key` со старого на новый по уликам, с записью в `meta.rekeyed_from`; (в) **инвариант-гейт: бамп `KEYER_VERSION` без сопровождающей re-key-миграции валит `smoke:memory-canon`**. Alias-таблица отвергнута (усложняет каждый read-path; батч-ре-кеинг проще и честнее).
- **Anki-диалект (`w:`-префикс) НЕ канонизируется** — он остаётся идентичностью ЭКСПОРТА (GUID стабилен). Маппинг anki→канон через существующий tag→note→body путь; при fan-out «один reviewId → несколько заметок одной lemma-tag-группы» item_key выбирается ДЕТЕРМИНИРОВАННО: заметка с pid-ключом приоритетна, иначе лексикографически минимальный канон-ключ.
- **Sentence-карты: контент-стабильный ключ** `sent:<text_key>#<order_index>#<template>` (тот же re-anchor-паттерн, что мигр. 060) — `card:<uuid>` ОТВЕРГНУТ: Pass 7 пере-генерит card id при импорте, история осиротела бы. Ключ переживает перенос и re-import.

### 3.4 Расширение `word_status` (мигр. 042, аддитивно, UPSERT-preserve)
```
srs_stability REAL, srs_difficulty REAL, srs_reviewed_at TEXT, srs_scheme TEXT
-- srs_scheme: NULL=sm2-lite (легаси) | 'fsrs-seed' | 'fsrs'
```
**Проекционный контракт легаси-колонок (MAJOR-фикс):** существующие `srs_due/interval/reps/lapses` НЕ замораживаются — на каждом FSRS-ревью пишутся из state: `srs_interval := (due − reviewed_at)/сутки`, `srs_reps/srs_lapses` из state → `rankByWeakness`, `ORDER BY srs_lapses`, leech-нудж, `getSrsSchedule` живут на свежих данных (класс save==paint закрыт). `srs_cards` FSRS-состояние — в `meta_json.fsrs` (без ALTER).

### 3.5 Судьба `srs_cards`
Остаётся: (а) слой linkage/export для Anki (GUID, exports-таблицы), (б) стор sentence-карт Trainer'а. Словарные note-карты Студии продолжают работать, но их планировщик становится fsrs-core (P3), а ревью пишутся в `review_log` под канон-ключом → память слова ЕДИНА между Залом и Студией (впервые).

### 3.6 Нумерация миграций (MAJOR-фикс R10)

## §4. Дизайн #2 — FSRS

### 4.1 Движок
- **`public/js/fsrs-core.js`** — UMD pure, без зависимостей, рукописный. Прецедент: `anki-apkg-core.js`.
- API (pure, nowMs-инъекция, детерминизм): `initState(grade, nowMs)` · `nextState(state, grade, nowMs)` · `retrievability(state, nowMs)` · `intervalFor(state, requestRetention)` · `dueAt(state, requestRetention)` · `seedFromSm2({interval, reps, lapses, ease?}, nowMs)` · `applyRow(state, logRow)` (ЕДИНАЯ функция шага — её используют и live-путь, и `replay(logRows)`; skip/seed-строки обрабатываются только здесь).
- `state = {stability, difficulty, reps, lapses, lastReviewedAt}`.
- **Робастность фолда (MAJOR-фикс):** `nextState` обязан быть определён на Δt≤0 (клампится в 0/same-day ветку); инжест клампит future-ts (зеркало future-clamp стрика `:2005`); tie-break порядка при равном `reviewed_at` — по `id` (детерминизм replay).
- **Golden-гейт `smoke:fsrs` с ЗАПИНЕННЫМ провенансом (MAJOR-фикс):** фикстура фиксирует {поколение FSRS — ОДНО, точный вектор весов, decay, same-day-формула, политика округления/клампа интервалов, requestRetention}; сгенерирована именованной референс-реализацией @точной версии (зафиксировать в README фикстуры); векторы обязательно включают same-day-цепочки, Again-тяжёлые последовательности, seed-ветки. Property-тесты: детерминизм; монотонность S по успехам; lapse снижает S; D∈[1,10]; Δt≤0 не взрывает; сид стабилен.
- **Контракт Again (MAJOR-фикс, продуктовая семантика):** grade 1 → **due = сейчас** (short-term/relearning шаг) — же-сессионный ре-тест провала, часть живого ощущения петли (`buildTrainSession.isDue` :1275), СОХРАНЯЕТСЯ. Отдельный golden-вектор именно на это; пункт №1 consumers-sweep P2.
- `requestRetention = 0.9`, веса — опубликованные дефолтные (D2).

### 4.2 Оценки
- Зал остаётся бинарным UX: верно→Good(3), неверно→Again(1), skip→kind='skip' grade 1 (фолд — как Again; из фита весов и retention-метрики skip ИСКЛЮЧЁН — «отказ отвечать» ≠ наблюдение памяти). FSRS корректно работает на подмножестве оценок. Hard/Easy — возможное будущее (по latency), НЕ сейчас.
- Студия-Trainer сохраняет 4 кнопки → 1..4 напрямую.
- Teach-показ (D5) не пишет ничего; немедленный пост-teach тест пишется с `meta.postTeach=1` (исключается из фита/Brier — успех из рабочей памяти).
- MC-канал: 25% chance-rate — в P6-калибровке отдельный бакет/chance-корректировка; решение о грейд-демоции MC — только после первого замера mc-vs-typed.

### 4.3 Миграция состояния (do-no-harm, R11)
- **Lazy-seed, без масс-ресчедула:** мигр. 042 только добавляет колонки. При СЛЕДУЮЩЕМ ревью слова: если `srs_scheme IS NULL` и есть легаси `srs_*` → записать `seed:<key>`-строку в лог (снимок+watermark) → `seedFromSm2` → обычный `applyRow`. Ни одно существующее due не двигается; откат тривиален.
- **`seedFromSm2` — спецификация вырожденных входов (MAJOR-фикс):** ветка `interval=0` (пост-lapse состояние — у активного профиля такие строки ВСЕГДА есть: lapse обнуляет и interval, и reps `:1907`) → эквивалент `initState(Again)` с documented S-полом; D — дефолт D0 (эвристика из reps/lapses ОТВЕРГНУТА: reps после lapse = длина текущей серии, не история — сигнал мусорный). Golden-вектор на ветку.
- **Shadow-diff перед переключением (MAJOR-фикс, measure-before-code):** пре-P2 харнесс `scripts/premium/srs-shadow-diff.js` — экспорт реального расписания владельца (бандл после P0) → симуляция N дней под nextSrs и под fsrs-core (обе математики pure) → отчёт diff: размер ежедневной очереди, распределение интервалов, слова с максимальным сдвигом due. Флип P2 — только после просмотра отчёта владельцем.
- **Студия-карты:** replay-сид из `srs_review_events` ТОЛЬКО если пре-фаза замера (§8 P·0) покажет, что история непустая и не изувечена cascade'ами; иначе — seedFromSm2 от interval_days/ease_factor. Не строим конвейер на неизмеренных данных.
- `nextSrs` (SM2-lite) остаётся в reader-morph как легаси-референс до конца P2, затем удаляется; переключение — атомарное в `checkTrainAnswer` (одна точка записи).
- **Смесь схем живёт неограниченно** (слова, которые больше не тренируются, остаются SM2-строками): retention-report печатает остаток legacy (`srs_scheme IS NULL AND srs_due IS NOT NULL`); опциональный migrate-on-serve (сид при попадании в очередь) — после обкатки, отдельным решением.

### 4.4 Оптимизатор — отложен (D2)
Дефолтные веса > SM2 по опубликованным бенчмаркам уже без оптимизации; оптимизатору нужно ~1000 ревью — при cap дневной цели 10 recalls это 3+ месяца накопления (честный темп). `review_log` с P0 копит именно тот датасет, который нужен. **Обязательная стратификация до любого фита:** по source×channel; исключаются kind='skip', `postTeach`, reading-tap (до порога надёжности §6); шкала Зала вырождена в {1,3} — фит на слитом логе без стратификации исказил бы Hard/Easy Студии.

## §5. Дизайн #3 — конфликт с Anki

**Принцип: сливаем ИСТОРИИ, не перезаписываем СОСТОЯНИЯ.**

- **Инжест (фильтры — MAJOR-фикс):** принимаются только type∈{0,1,2} (learn/review/relearn; manual-reschedule type=4 с ease=0 и cram — drop со счётчиком) и ease∈1..4; `reviewed_at` из Anki-ревью (не `new Date()`); future-ts клампится. meta несёт `{anki_card_id, ord, anki_type}` — направление карты в провенансе. id = `anki:<reviewId>` из самого ревью.
- Состояние слова = `replay()` слитого лога (упорядочен по ts, tie-break id; Δt≥0 по контракту §4.1 — clock-skew чужого устройства не взрывает фолд). Ревью в обоих местах оба считаются — это правда о практике; существовавший гард local-newer СНИМАЕТСЯ ОСОЗНАННО: его роль (не дать чужим часам затереть свежий локальный прогресс) выполняют клампы + упорядоченный фолд (состояние теперь не перезаписывается, а выводится).
- **Recompute fan-out — явный компонент (MAJOR-фикс, не «бесплатно»):** после инжеста N ревью, сгруппированных по M словам: для каждого item_key → `replay` его лога → запись через **выделенный `updateSrsState(itemKey, state)`** — srs-only UPDATE (НЕ `setWordStatus`: тот требует status и перезаписывает его; якоря — COALESCE). Этот writer вносится в gate-consumers-sweep как ВТОРОЙ писатель `srs_*` (урок multitab). Стоимость: O(строк лога item'а) на слово, батч в одной транзакции; фикстура «anki-ревью сегодня → слово уходит из due Зала» — в `smoke:memory-canon`.
- Существующий гард `anki_managed` (исключение из in-app due `srs_cards`) СОХРАНЯЕТСЯ — анти-двойное-ревью на уровне карт Trainer.
- **Каннибализация очереди Зала recognition-ревью Anki** (R2: fan-in «3 Anki-карты одной леммы → серия Good → S взлетает → слово уходит из due Зала → production-тир не практикуется») — адресуется развилкой D9 (стратификация/двухтрековость), плюс P6-разрез recall-rate по source ловит расхождение числом.
- **Remote-пользователь без read-back:** ничего не меняется — экспорт one-way, две независимые системы, документируем честно. Никаких обещаний mobile-sync (вердикт Q3 2026-06-18 в силе).

## §6. Дизайн #6 — reading-native retrieval (моат)

Фундамент готов: due-множество известно (`getSrsSchedule`), `decorateWords` сканирует всё, ключ-парити решён (единый кейер §3.3).

1. **Тихий due-маркер:** слово из due-множества получает модификатор `.rm-w-due` — деликатный знак (точка/кольцо), НЕ новый фоновый тинт. **Только exact-уверенные резолюции** (BLOCKER/MAJOR-фикс R10: гомограф בקר-класса не маркируется и не грейдится; неуверенно разрешённая форма — тихо пропускается, счётчик подавленных маркеров идёт в retention-report как мера потерь recall-возможностей).
2. **Reveal-then-grade карточка (BLOCKER-фикс B7, R2):** тап по due-слову → карточка в режиме «вспомни»: глосс скрыт → пользователь пытается про себя → «Показать ответ» → глосс виден → две кнопки **«Знал (3) / Не знал (1)»**. Оценка пишется ТОЛЬКО ПОСЛЕ показа ответа (канонический flashcard-флоу; «grade-до-reveal» ОТВЕРГНУТ как измерение иллюзии знания — Koriat & Bjork). Закрытие карточки без оценки не пишет ничего, но инкрементит локальный счётчик shown-vs-graded → **reading-tap включается в калибровку/фит только при abandonment ниже порога** (MNAR-смещение под контролем, не только «задокументировано»).
3. **Ручной уровень reading-тап НЕ двигает** (по умолчанию; развилка D8): связка recall→nextLevel утверждалась для тестируемых каналов (MC против морфо-честных дистракторов, typed против accepted-skeletons); самоотчётное узнавание при видимом слове, дотапывающее слово до «known» и ретирующее его из i+1/сессий/production-тира — отвергнуто. Расписание (FSRS) двигается; level — нет.
4. **Пассивная встреча ≠ ревью:** скролл мимо due-слова не пишет НИЧЕГО (анти-паттерн LingQ «paging→known» сознательно отвергнут).
5. **Ротация якоря (MAJOR-фикс R2):** open-text сессии выбирают occurrence round-robin по reps (`occ` уже собирается `_scanWords`); cross-text — предпочтение ДРУГОГО предложения того же текста через order_index-скан от сохранённого якоря. Иначе слово 8 раз видит идентичный клоуз → S меряет память предложения (encoding specificity), не знание слова.
6. **study_day:** в `recalls` считаются только пост-reveal оценки (reading-тап «Знал» без показа не существует по построению п.2) — дневная цель/стрик не закрываются бесплатными тапами.
7. **Пре-текст варм-ап (follow-up P5.1, развилка D4):** «в этом тексте N слов к повторению — повторить до чтения?» → мини-сессия на предложениях ЭТОГО текста.
8. **Конструкт-дрейф recognition-vs-production:** P6 обязан резать observed-recall по source; порог прекоммичен ДО ship P5: если recall-rate(reading-tap) − recall-rate(room-recall) > 15 п.п. на равных интервал-бакетах → грейд reading-тапа демотируется до Hard(2) или вводится отдельный весовой профиль (данные решают, порог зафиксирован заранее).
