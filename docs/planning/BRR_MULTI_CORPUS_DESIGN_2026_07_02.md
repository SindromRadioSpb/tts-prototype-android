# Мульти-корпусный Читальный зал — дизайн-варианты · 2026-07-02

**Статус: ПРЕДЛОЖЕНИЕ → выбор владельца.** Контекст: владелец переформулировал Эпик B —
«Корпус» сегодня на деле «Корпус библиотеки Бен-Иегуды»; в перспективе корпусов много
(«Материаловедение», другие предметные), и «Мои тексты» должен стать **полноправным корпусом**,
продуктово зрело — это отличительная фишка программы. Полка v3.11.69 — первый шаг, не финал.
Роли: R3 (вед., архитектура) · R4 (премиум-UX) · R5 (рынок) · R6 (куратор) · R8 (graded) · R9 (провенанс).

## 1. Фундамент, общий для ВСЕХ вариантов поверхности (R3)

Сегодня «Корпус» — это захардкоженный Бен-Иегуда: каталог `corpus-catalog-vN.json`, дрилл
Период→Автор→Работа, FTS, бейк-пайплайн. Масштабирование = **реестр корпусов + контракт
провайдера**, за которым каждый корпус — плагин:

```
CorpusRegistry: [ CorpusManifest ]
CorpusManifest {
  id, title, icon, cover, description,          // идентичность (витрина)
  kind: 'baked-catalog' | 'local-db' | ...,     // адаптер
  taxonomy: [ {key:'period',label:…}, {key:'author'}, … ]   // РОДНОЙ дрилл корпуса
           // Бен-Иегуда: период→автор→работа · Мои тексты: фасеты уровень/теги/источник
           // Материаловедение (буд.): раздел→тема→текст
  capabilities: { translations, overlays, audio, fts, bands, … }   // ЧЕСТНАЯ витрина (R9)
}
CorpusAdapter (контракт): home() · drill(level, key) · search(q) · counts() · openText(ref)
```

- **BakedCatalogAdapter** — нынешний Бен-Иегуда-код, обёрнутый контрактом; будущие предметные
  корпуса = новые каталоги JSON через ТОТ ЖЕ адаптер (продюсер каталога параметризуется).
- **LocalDbAdapter** — «Мои тексты»: живой OPFS-запрос (корпус динамический — растёт из Студии),
  фасеты из родных мета (уровень/теги/источник/прогресс), клиентский поиск.
- **Правило двух:** абстракцию доказываем ВТОРЫМ корпусом («Мои тексты»), не строим«на вырост»
  больше контракта; «Материаловедение» потом ложится дёшево.
- Кросс-корпусное остаётся кросс-корпусным (один localDb): статусы слов, SRS, Continue,
  закладки — **словарь ученика следует за ним между корпусами** (это и есть моат, R2/R5).

### Глубина «Мои тексты» как корпуса (анти-заглушка, R8 — ключевое)
1. **i+1 и покрытие словаря на СВОИХ текстах**: движок per-text coverage known/learning/new +
   frontier уже существует (autogen Stage 4, Studio) — переиспользовать в карточках корпуса
   «Мои тексты» → сортировка «по лёгкости для меня», как в корпусных полосах. Уникально на рынке:
   graded-reading по СОБСТВЕННОМУ контенту.
2. **Аудио/караоке**: у студийных текстов есть TTS-аудио → reader-караоке Зала работает сразу.
3. **Фасеты**: уровень · теги · источник · статус (не начат/читаю/прочитан) · сорт (недавние/
   алфавит/по лёгкости).
4. **Воронка**: CTA «+ Добавить текст» (→ Студия; позже — PDF-импорт эпик C) прямо в корпусе.
5. **Честные бейджи возможностей** (R9): «перевод: свой» vs «перевод: предвыч.», «морфология:
   офлайн», «точный режим: live» — без имитации того, чего у корпуса нет.

## 2. Варианты ПОВЕРХНОСТИ (что видит пользователь)

### A. «Полки+» — статус-кво расширяем
Один корпус-таб (Бен-Иегуда), каждый новый корпус = ещё одна полка на его home.
- R4: дёшево; но идентичность корпусов растворяется, home превращается в ленту полок.
- R5: НЕ читается как «фишка»; выглядит как надстройка. **Отклонить как целевое** (полка
  v3.11.69 остаётся как быстрый доступ, см. гибрид).

### B. «Библиотека корпусов» — хаб-уровень L0
Таб «Корпус» становится «Библиотека»: L0 = **витрина корпус-карточек** (обложка/иконка,
описание, счётчики «796 готово из 26 415», бейджи возможностей, персональный прогресс в
корпусе, для «Мои тексты» — CTA добавления). Тап → родной home корпуса (его дрилл + его rails).
Сверху L0 — кросс-корпусные rails (Продолжить чтение — из ЛЮБОГО корпуса).
- R4: сильная идентичность, витрина = премиум; +1 уровень навигации (тап до текста дольше).
- R6: куратор получает место рассказать, ЧТО это за корпус (провенанс, объём, назначение).
- R5: расширяемо на витрине видно будущее («скоро: тематические корпуса») — продаёт роадмап.

### C. «Свитчер-линза» — переключатель корпуса на месте
Home Корпуса получает премиум-пилюлю «⟨🏛 Бен-Иегуда ▾⟩» — переключение подменяет провайдера
под ТЕМ ЖЕ хромом (поиск/rails/дрилл перерисовываются в родную таксономию выбранного корпуса).
- R4: минимальная глубина, мгновенное «переключение линзы»; но идентичность корпуса сжата до
  пилюли, витрины нет; onboarding нового корпуса слабее.
- R3: тот же реестр/адаптеры; чисто поверхностное отличие от B.

### D. Таб на корпус
«Мои» как четвёртый таб. Не масштабируется на «много корпусов» по определению задачи владельца.
**Отклонить.**

### ⭐ Рекомендация: гибрид B+C («витрина + линза»)
- **L0-хаб** (вариант B) — идентичность, витрина, вход и onboarding корпусов;
- **внутри корпуса** заголовок-хлебная-крошка «Библиотека › 🏛 Бен-Иегуда ▾» — та же пилюля-
  свитчер (вариант C) для мгновенного бокового перехода без возврата на L0;
- полка «Мои тексты» на home Бен-Иегуды (v3.11.69) **сжимается до мини-rail с «→ весь корпус»**
  (быстрый доступ сохранён, дублирование убрано);
- кросс-корпусный Continue — на L0; внутрикорпусные rails — в корпусах.
Это паттерн зрелых читалок (библиотека → полка → книга) + скорость power-user'а. Фазируется:
Ф1 реестр+адаптеры+хаб L0+«Мои тексты»-корпус (фасеты+поиск+CTA) · Ф2 свитчер-пилюля +
i+1-coverage на своих текстах · Ф3 generic-дрилл для предметных каталогов (когда появится
первый реальный, напр. «Материаловедение»).

## 3. Инварианты
Room-only (index.html не трогать) · byte-parity reader един для всех корпусов · честность
возможностей per-corpus (R9, никаких фейковых бейджей) · один localDb = кросс-корпусный
словарь/SRS · большие профили: адаптеры сами выбирают перф-стратегию (26K каталог ≠ 500 своих
текстов) · i18n ×3 · гейты: mytexts-smoke расширяется до corpus-hub-smoke.
