# Library export — перф-трек P (recon-дизайн)

**Дата:** 2026-06-24 · **Статус:** RECON — awaiting owner go · **Эпик:** [[LIBRARY_EXPORT_PROGRESS_T2]] (P → потом T3).
**Цель:** сократить время экспорта аудио для больших библиотек (владелец: 169 текстов / 8906 клипов).
Роли: **R5** (offline-first, реальное использование) · **R3** (архитектура) · прод-стабильность (CX23: 1.5 CPU, 1536 МБ).

---

## 0. Факты разведки (что есть на самом деле)

- **Аудио-байтов локально НЕТ** — ни OPFS (только метаданные `audio_assets`: `migrations.js:60-71`), ни SW-кэш
  (`sw.js:235-237` — `/api/*` network-only, явно не кэшируется), ни Cache API. → идея «читать из OPFS» НЕВАЛИДНА.
- **Cache-HIT** обслуживает браузерный HTTP-кэш: сервер шлёт `Cache-Control: public, max-age=31536000, immutable` + ETag
  (`server.js:2729+`). Повторные фетчи одного `assetKey` — из HTTP-кэша / 304, без сети.
- **Cache-MISS** (холодный браузер — типично для устройства-ИСТОЧНИКА при переносе) = сеть + на сервере per-request:
  `fs.statSync` + **DB-lookup `relative_path` + `touchAudioAsset` UPDATE** (`server.js:2737-2756`). Это и есть ~850 мс/клип
  и нагрузка на прод-БД при росте параллелизма.
- **Параллелизм = 6**, без обоснования (`index.html`, `v3ExportRunLocal`/`CONCURRENCY`). Prefetch — серверная генерация,
  локальный клиентский кэш НЕ наполняет (для экспорта бесполезен).
- Прод за Traefik (HTTP/2 ⇒ мультиплексирование многих параллельных запросов по одному соединению — рост параллелизма дёшев на проводе).

**Вывод:** «премиум» здесь = (1) выше параллелизм для cache-miss; (2) «тонкий» серверный путь, чтобы miss был дешевле И
безопаснее при параллелизме. Выигрыш растёт с долей miss — а именно cold-источник при переносе на другое устройство.

## 1. Дизайн P

### P-server — «тонкий» bulk-путь отдачи аудио (предусловие для безопасного параллелизма)
- **P-S1.** `/api/audio/:assetKey?bulk=1` (или заголовок `X-Bulk: 1`): пропустить DB-lookup и `touchAudioAsset`;
  резолвить файл по детерминированному пути `audio-cache/<assetKey>.mp3` (фолбэк, который и так уже есть);
  чистый `fs.createReadStream().pipe(res)` + те же immutable-заголовки. Read-only, stateless, идемпотентно.
- **Инвариант:** поведение для обычных (не-bulk) запросов НЕ меняется. `touchAudioAsset` (телеметрия «last_used») для
  bulk-экспорта осознанно пропускается — это пакетная служебная операция, не пользовательское воспроизведение.
- **Эффект:** убирает per-request DB-запись → 8906 запросов больше не бьют по прод-SQLite; статичный стрим масштабируется параллелизмом.

### P-client — параллелизм + тонкий URL (в `v3ExportRunLocal`)
- **P-C1.** Поднять `CONCURRENCY` 6 → **12** (консервативно; HTTP/2 мультиплексирует, файлы ~40 КБ, серверный путь теперь без БД).
  Вынести в именованную константу с комментарием-обоснованием (чего не было).
- **P-C2.** Экспорт фетчит `…?bulk=1`. Cache-HIT всё равно из HTTP-кэша (immutable не зависит от query — но query меняет ключ кэша!
  ⚠ см. риск R-1) — поэтому решить вопрос ключа кэша (вариант: не менять URL, а слать заголовок `X-Bulk`, чтобы ключ HTTP-кэша
  совпадал с обычным воспроизведением и hit'ы переиспользовались). **Рекомендация: заголовок `X-Bulk: 1`, не query-параметр.**
- **P-C3.** (опц.) Адаптивно: стартовать с 12; если идёт серия cache-hit (быстрые ответы) — не важно; если miss-латентность
  высокая и нет ошибок — можно не трогать. Держим просто: фикс 12. Замер решит, нужен ли тюнинг.

### НЕ входит в P (возможный P2, только если замер покажет, что round-trips всё ещё доминируют)
- Батч-эндпоинт (POST список ключей → один ответ tar/multipart): убирает N round-trips, но +сложность и грубее прогресс.
  HTTP/2 уже мультиплексирует — сначала меряем P-C1/P-S1, потом решаем.

## 2. Стыковка с T3 (чтобы не переделывать)
- Параллелизм (P-C1) и тонкий URL/заголовок (P-S1/P-C2) — это пара «константа + способ запроса». Обе тривиально переносятся
  в Web-Worker-цикл T3. P НЕ строит ничего, что T3 снёс бы. Порядок P→T3 безопасен.
- OPFS-источник в T3 тоже отпадает (аудио не в OPFS) — T3 = Worker + File System Access streaming, фетч по сети остаётся.

## 3. Критерии приёмки
1. `?bulk`/`X-Bulk` путь отдаёт тот же байт-контент, что обычный (одинаковый clip), immutable-заголовки на месте; обычный путь не задет.
2. Экспорт с аудио работает как раньше (карточка/прогресс/отмена/чип/manifest) — это рефактор скорости, не поведения.
3. На прод-профиле владельца (8906): измеренный темп выше прежних ~7–16/сек (особенно на cold-источнике); прод healthz зелёный,
   без всплеска 5xx/латентности под параллелизмом 12.
4. HTTP-кэш cache-HIT переиспользуется (ключ кэша не разъехался — отсюда выбор заголовка, а не query).
5. Гейты: `smoke:i18n`, `smoke:reader-parity`, синтаксис; `test:api-smoke` (затронут server.js!); SW bump.

## 4. Риски
- **R-1 (ключ HTTP-кэша):** query-параметр `?bulk=1` создаёт ОТДЕЛЬНЫЙ ключ кэша → cache-hit'ы из обычного воспроизведения
  не переиспользуются (всё станет miss!). **Митигировать заголовком `X-Bulk: 1`** (+ `Vary` не нужен, контент тот же).
  Альтернатива: оставить URL, но это ухудшит, а не улучшит — поэтому заголовок.
- **R-2 (прод-нагрузка):** параллелизм 12 × стрим — лёгкий после снятия БД-записи; healthz/латентность под наблюдением при верифи.
- **R-3 (server.js трогаем):** прод-эндпоинт — обязательный `test:api-smoke` + прод-верифи Node-fetch (не Windows curl, см. [[feedback_curl_utf8_egress_myth]]).

## 5. Фазы (каждая — гейты + прод-верифи, commit+push по умолчанию)
- **Ф1 (server):** P-S1 тонкий `X-Bulk` путь + api-smoke.
- **Ф2 (client):** P-C1 параллелизм 12 + P-C2 `X-Bulk` header в `v3ExportRunLocal`; прод-замер на профиле владельца (Kapture).
