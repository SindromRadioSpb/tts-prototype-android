## 1. Диагноз (recon по живому коду)

- Хранилище УЖЕ общее: материализованные корпус-работы лежат в той же таблице `texts`
  (дискриминатор `source_meta_json.corpus`). **v3-библиотека уже листит их** (`listTexts
  {limit:200}` @index.html:23478, без корпус-фильтра) — т.е. сегодня они «видимы», но:
  1) смешаны со своими без провенанс-бейджа; 2) при сотнях материализованных работ **вытесняют
  свои тексты из limit-200**; 3) несут полный набор действий, опасных для канона: «Удалить»
  (сносит локальную копию канона), «Изменить» (правка TITLE = клоббер канонического заголовка),
  «Обогатить заново» (**перезаписал бы курируемый baked-перевод личным Gemini — R11**).
- `manual_smart_tag` (🔥/✓ override) — колонка `texts`; модал пишет колонки строки напрямую.
- Re-import: корпус-работа переимпортируется `importBundle(mode:'skip')` (правки строки
  выживают), но re-shard каталога v(N+1) может заменить работу → правки канон-строк хрупки.

## 2. Принципы (инварианты решения)

- **R9 derived≠asserted:** канонические поля (title, source_label, тело) — read-only с бейджем
  «🏛 канон Бен-Иегуды»; ЛИЧНАЯ мета (tags, level, тема, smart-override) — редактируемый
  пользовательский слой.
- **R11 do-no-harm:** «Обогатить заново» на каноне — заблокировано с честной причиной (v1);
  «Удалить» → семантика «Убрать с устройства» (работа снова откроется из каталога Зала).
- Правки index.html легитимны (это Студия-фича; инвариант «не трогать» был про работу над
  Залом) — но: CSS-ловушки CLAUDE.md, SW bump, скрин @380px, полный прогон smoke.
