## 1. Что это и зачем (без изменений)
Полка «Следующий для тебя»: из ~26 455 публичных ивритских работ рекомендуем те, что в зоне Krashen
i+1 (знаешь ~80–95% слов работы, остаток — посильный шаг роста). Превращает каталог в управляемую
траекторию чтения. Замыкает learner-loop Зала: читай→тап→карточка→Сохранить→цвет-статус→**что
читать дальше**→рост. Рыночный эталон — LingQ/LingoClip (они это и продают).

## 2. Архитектурная находка рекона (решает D1)
**Бейкнутые работы НЕ несут пословной морфологии** — в каждой строке только `hebrew_plain /
hebrew_niqqud / russian`. → vocab-профиль НАДО строить лемматизацией на build-time. Офлайн-путь =
form-first над Pealim-словарём (`pealim-infl-v12`, 9279 парадигм, 226 834 cell-форм → 145 014
различных консонантных форм). Ключ join'а — канонический lemmaKey: **`pid:<pealim_id>`** иначе
`norm(lemma|word)#pos` (ниqqud снят) — **байт-идентичен** `getKnownWordStates` (`local-db.js:2114`)
и `NotesAutoGen.lemmaKey` (`notes-autogen.js:355`). Профиль работы должен жить в ЭТОМ пространстве.

## 3. ЗАМЕР (честный, перепроверенный) — на 796 бейкнутых работах

> ⚠ **Скос выборки (R10-честность):** 796 локальных работ по полю `corpus.era` = **696 без эпохи +
> 59 haskalah + 41 tehiya; 0 modern, 0 mandate** (метка бейк-таргета «modern/mandate/unknown» — это
> Wikidata-эпоха из манифестов, а НЕ поле тела работы). Это **30.6%** корпуса; непокрытые **69.4%**
> (tehiya 38% · medieval 20% · haskalah 11%) — самые архаичные/именословные. Все числа ниже — на
> «лёгкой» трети, это ПОЛ, не потолок сложности.

### 3a. Размер сайдкара (РОБАСТНО — не зависит от learner-прокси)
| | 796 работ (замер) | 8099 bakeable (экстрап.) | 26 455 полный (экстрап.) |
|---|---|---|---|
| packed delta-varint gz | **292 КБ** (dict 166 + payload 125) | ~1.65 МБ | **~4.7 МБ** |
| наивный JSON gz | 544 КБ | ~5.4 МБ | ~17.7 МБ |

- Offline Pealim match-rate: **86.7%** токенов → `pid:` лемма (13.3% fallback). По эпохам стабильно:
  unknown 86.2% / tehiya 89.2% / haskalah 89.3% (архаика НЕ роняет, но biblical/medieval не тестилось).
- Различных лемм/работа: min 2 · median 155 · mean 302 · p90 734 · max 3203.
- **Общий словарь BOUNDED:** Heaps V≈1968·N^0.187 (β=0.19, сублинейно), упирается в потолок 9279
  парадигм → shared-dict НЕ растёт ×N, насыщается. Растёт только дешёвый сжимаемый fallback-хвост.
- **Честная сверка масштаба** (adversary): живой `corpus-search-v7` ≈ **530 КБ gz на проводе**.
  «17.7 МБ naive-JSON» — strawman; честно: vocab ~4.7 МБ packed-gz ≈ **9× самого крупного ленивого
  ассета Зала**. Отгружаемо при ленивой загрузке по первому тапу — но **нужен hard size-gate** в
  сборке (fail при >X КБ/1000 работ), иначе будущее bake-расширение тихо перейдёт mobile-обрыв.

### 3b. Качество i+1 — НЕ валидировано (главная честная поправка)
Симуляция «учащийся знает top-K частотных лемм» под **4 комбинациями** знаменатель×вес @K=2500:

| знаменатель | вес | zone(80–95%) | tooEasy | tooHard | median |
|---|---|---|---|---|---|
| matched-only | type | **647** | 70 | 61 | 0.884 |
| matched-only | token | 604 | 157 | 17 | 0.927 |
| **all-token** | type | **148** | 3 | **627** | 0.732 |
| **all-token** | token | 411 | 4 | 363 | 0.804 |

**Три перепроверенных дефекта прежнего «зона валидна»:**
1. **Циркулярность (Zipf-голова на себе):** top-2500 лемм = **93.0%** всех matched-токенов ПО
   ПОСТРОЕНИЮ (top-200=50% · top-1000=78% · top-2500=93% · top-4000=98%). «Сладкая точка» — свойство
   Zipf-распределения иврита, проверенное против собственной головы корпуса. Нет held-out, нет
   реального профиля. Реальный known-set учащегося — комковатый и с более редким хвостом → истинное
   покрытие НИЖЕ и разбросаннее гладкой кривой.
2. **Знаменатель решает всё:** matched-only (647 в зоне) против all-token (148 в зоне, 627 too-hard)
   — ОДНИ И ТЕ ЖЕ данные, противоположный вывод. matched-only отвечает «могу ли сделать карточки на
   разрыв», all-token — «могу ли ПРОЧИТАТЬ прозу». Рекон мерил первое, фича обещает второе.
3. **Был type-, а не token-coverage:** прежняя кривая пересекала РАЗЛИЧНЫЕ множества лемм, не
   взвешивая по частоте токенов. «Валидированная token-покрытие-кривая» не существовала.
4. **Гетерогенность fallback:** median 12.5% · p90 18.8% · **max 68.1%**; **101/796 (12.7%) работ
   >18% fallback-нагрузки** — matched-only там сильно льстит читабельности.

**Что РОБАСТНО выжило:** (a) сайдкар отгружаем (байтовый факт); (b) только A ранжирует по корпусу
(логическая необходимость, не замер); (c) **пустая зона на холодном старте** (K=200/500 → 0 в зоне,
всё too-hard) — держится при ЛЮБОМ знаменателе (пустое пересечение пусто) → двух-рельсовый on-ramp
обязателен; (d) корпус «выпускает» работу too-hard→в-зоне→too-easy с ростом — направление верно.

## 4. Решения D1–D4 (рекомендации + adversary-поправки)

**D1 — как строить профиль → A (с предусловиями).** Сайдкар несёт ИНГРЕДИЕНТЫ, не вердикт:
(1) общий словарь `{lemmaKey→int id}` назначен **стабильной сортировкой** lemmaKey (не порядком
хеш-итерации); (2) per-work delta-varint списки id; (3) **per-lemma token-counts** (не presence-бит);
(4) per-work matched-count + total-token-count; (5) **model-id лемматизатора** (`pealim-infl-v12`) в
заголовке. Покрытие-% ВСЕГДА пересчитывается клиентом против живого `getKnownWordStates()` на рендере
— замороженный % в payload ЗАПРЕЩ�ён (мгновенно протухает после первого Сохранить).
*Adversary:* продюсер ОБЯЗАН импортировать ТОТ ЖЕ `notes-autogen.js` lemmaKey-кор (не реimplement);
**switch-to-C триггер:** size-gate сработал ИЛИ #pos-parity-gate не зелёный ИЛИ churn заметно
переставляет полки → падаем в C (лёгкий ranking-сайдкар + on-device пересчёт покрытия).

**D2 — «знаю» + зона → ДВА числа на работу, никогда одно.** (1) i+1 DRILL-покрытие по **matched
(learnable)** лексике, **token-взвешенно**, «знаю» = known+learning (most-advanced-wins, как уже в
`getKnownWordStates`); питает loop тап→карточка→статус→Anki. (2) Честный сигнал READING-LOAD =
all-token покрытие + флаг «много имён/архаики» при высокой fallback-доле (провизорно >18% или
matched<50%). Полоса **80–95% — РАБОЧАЯ ГИПОТЕЗА, не факт**, до real-profile+token замера (§7).
*Adversary:* смоук-гейт обязан валиться, если два знаменателя расходятся выше порога, а UI это не
показывает — «одно число, которое лжёт» не должно дойти до бейджа.

**D3 — холодный старт → двух-рельсовый on-ramp, НИКОГДА не пустой.** Рельс 1 = персональная
«Следующий для тебя» (i+1-пересечение), рендерится ТОЛЬКО когда ≥N работ реально в зоне — никогда
ярлык «для тебя» над too-hard-списком. Рельс 2 = безпрофильная «С чего начать» по ВНУТРЕННЕЙ лёгкости
(короткие + концентрация частотной головы — бесплатно из сайдкара), **НЕ по эпохе** (87% бейк-среза
без эпохи — эпоху не на чем считать). Тот же премиум-компонент, другой заголовок+провенанс, на
безпрофильных работах БЕЗ %-бейджа (87% против пустого known-set — ложь; показываем «короткий ·
частотная лексика · есть озвучка»). Рельс 2 угасает, Рельс 1 наполняется с ростом; на верхнем конце
движок переформулирует в «следующий вызов». *Compromise:* «короткий» надо гейтить мин. связной
длиной + частотной концентрацией (кратчайшие в корпусе = стихи/фрагменты/архаичные обрывки).

**D4 — версия/публикация → lockstep в каталог.** Vocab-сборка ВНУТРИ `build-corpus-catalog --full`
(тот же проход, НАСЛЕДУЕТ версию V — сейчас v7, свой счётчик ЗАПРЕЩён) + в гейт-сет
`publish-corpus-batch` + selfCheck → не рассинхронится. Шип ПЛОСКИМ сперва (1 shared-dict + 1
payload, `?v=V`, force-cache, НЕ precache, SW CACHE_VERSION бамп каждый републиш). Per-era шарды —
отложить до измеренной нужды (i+1 кросс-эпошный → преждевременный шардинг заставит тянуть все шарды).
Покрывает ТОЛЬКО бейкнутые (небейкнутую нельзя открыть). *Adversary:* зафиксировать инвариант
**«vocab-сайдкар = только public-domain»** СЕЙЧАС (до появления BYOK/премиум-текстов — их
лемма-карту нельзя класть в публичный ассет).

## 7. СЛЕДУЮЩИЙ ЗАМЕР (gate до любого producer-кода) — real-profile validation

GZIP-размер для hard build-gate (база: corpus-search-v7 ≈530 КБ gz). **Если зона на реальном профиле
схлопывается как all-token (напр. <50 читабельных работ у типичного учащегося) → A переусложнён,
падаем в C.** Это и есть A-vs-C-решатель. Producer-кода НЕТ, пока не пройдёт.
