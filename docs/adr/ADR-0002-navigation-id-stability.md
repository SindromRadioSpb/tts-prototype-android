# ADR-0002 — Navigation ID stability: order_index ≠ id (stable ids only)

Статус: Accepted  
Дата: 2026-01-18  
Patch ID: ADR-NAV-ID-STABILITY-01  
Владелец: Navigation / Premium PRO

---

## 1) Контекст и проблема

В домене Navigation/Search есть риск критической путаницы:
- перепутать “стабильный идентификатор сущности” (textId/sentenceId/noteId/rowId) с “позиционным индексом” (order_index / orderIndex);
- начать использовать позиционные поля как deep link id или ключи резолва.

Это ломает:
- стабильные deep links (ссылка перестаёт указывать на ту же сущность),
- детерминизм NAV acceptance tests,
- воспроизводимость поведения для агентов,
- “Back to results” (указатель активного hit может указывать на другой элемент после изменений в тексте).

---

## 2) Решение (Decision)

1) **Для любых навигационных сущностей используются только стабильные IDs**:
   - textId / sentenceId / rowId / noteId / audio_id (в терминах DB_SCHEMA).
2) **Позиционные значения (order_index / orderIndex) НЕ являются id** и **никогда** не используются:
   - как Target.id,
   - как deep link id,
   - как ключ резолва сущности в БД.
3) `orderIndex` допускается только как:
   - UI-лейбл (“Строка № …”),
   - вспомогательный хинт для позиционирования/скролла (если stable id уже известен),
   - диагностическое поле в результатах поиска (nullable).

---

## 3) Нормативные правила (Normative rules)

### 3.1. Что такое “stable id”
Stable id — это идентификатор, который:
- не меняется при редактировании текста,
- не зависит от вставок/удалений строк,
- сохраняется при пересортировках/фильтрациях/перегенерации выдачи,
- допускает долговечные ссылки (deeplink) и резолвится в NOT_FOUND корректно.

### 3.2. Что такое “positional index” (order_index)
Positional index — это “номер/порядок” в рамках текста.
Он может меняться при:
- вставке/удалении строк,
- восстановлении/пересборке текста,
- изменении сортировки/структуры.

Следовательно, positional index:
- полезен в UI (показать порядок),
- опасен как id (не гарантирует стабильную адресацию).

### 3.3. Правило Target.id
В Target v=1:
- `type=text`   => `id = textId`
- `type=sentence` => `id = sentenceId`
- `type=note`   => `id = noteId`
- `type=search` => `id = opaque searchKey` (НЕ entity id)

Запрещено:
- класть order_index/orderIndex в `id` для любых типов.

### 3.4. Где допускается orderIndex
Допускается:
- в Search results DTO как поле отображения;
- в SearchSession как “подсказка”, но не как источник истины;
- в UI data-attrs как вторичное значение (не ключевое).

Запрещено:
- строить deep link из orderIndex,
- резолвить сущность по orderIndex,
