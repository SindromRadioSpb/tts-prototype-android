# R11 (регрессолог do-no-harm) — адверсариальная критика retention-recon (сырой вердикт, 2026-07-02)

Все пункты мандата проверены по коду; ниже — только выдержавшие проверку (два кандидата из мандата отброшены после проверки: (а) рассинхрон бейджа D2-CTA и очереди при lazy-seed не возникает — и `dueCounts`, и `getDueWithSource`, и `getDueReviewCount` читают одну и ту же хранимую колонку `srs_due`, которую lazy-seed не трогает до ревью; (б) «skip станет жёстче по расписанию» в чистом виде не подтверждён — сегодняшний reset interval=0 сам по себе максимально жёсток; выжившая часть этой претензии — про permanenтный D-штраф и загрязнение лога — вошла в M3).

---

**1. BLOCKER — Канон-инвариант «состояние пересчитываемо из лога» ложен для всех seeded-слов; independent-oracle гейт либо всегда красный, либо беззубый на месяцы.**
Обоснование: §3.2 recon явно запрещает фабрикацию истории («Для word_status.srs_* истории НЕТ — не фабрикуем»), §4.3 сидирует состояние через `seedFromSm2` вне лога, а §7 требует `replay(review_log) == накопленное состояние`. Для любого слова, тренированного до P0 (у владельца ВСЕ активные слова — srs-строки копятся с v3.11.32), replay от пустого лога ≠ seed+пост-seed-ревью — навсегда. Хуже: между P0 и P2 лог УЖЕ пишется, а SM2-колонки продолжают обновляться → seed на P2 «поглощает» P0–P2-ревью, и replay(seed + весь лог) применит их дважды. Исключить seeded-слова из гейта = на реальном профиле гейт первые месяцы не проверяет ничего (D3-замер: 5124 статуса, весь живой словарь — pre-P0).
Fix: материализовать seed как строку лога (`source='seed'`, grade NULL, `meta`={sm2-снимок, provenance, watermark-ts}) — это не фабрикация, а честная запись реального события «состояние сидировано из SM2 в момент T» (R9: derived помечен как derived); `replay()` обязан стартовать item от seed-строки и игнорировать строки ≤ её watermark. Тогда оракул самодостаточен от сырого входа с первого дня.

**2. BLOCKER — Схема id `review_log` ломается на бандл-merge: per-device монотонный id даёт тихую ПОТЕРЮ чужой истории, а производный от `srs_review_events.event_id` — ДУБЛИРОВАНИЕ.**
Обоснование: §3.2 задаёт `'app:<монотонный id>'` + merge `INSERT OR IGNORE по PK` как блокер-требование. Монотонный счётчик не уникален между устройствами: `app:17` устройства B столкнётся с `app:17` устройства A (разные ревью) → чужая строка молча выброшена из «таблицы-истины». Обратная схема (`app:sre:<event_id>` для бэкфилла) тоже не работает: `importBundle` Pass 8 генерит `srs_review_events.id` заново через `uuid()` (local-db.js:5059–5062) — id НЕ стабильны между устройствами и даже между двумя импортами одного бандла (комментарий «idempotent via id PK» ложен: fresh uuid никогда не конфликтует → re-import дублирует всю историю ревью на смёрдженную Pass 7 карту). Тот же uuid()-паттерн у events Pass 12 (:5135) уничтожает `anki:<reviewId>`-идентичность events-строк на мигрированном устройстве.
Fix: id = контент-детерминированный (`hash(item_key|reviewed_at|grade|source)`) либо `app:<installation-nonce>:<counter>`; бэкфилл Anki — по `payload_json.anki_review_id` (переживает импорт), не по `events.id`; в P0 заодно починить Pass 8/12 — переносить исходные PK (они уже уникальны) вместо uuid(), иначе P3 «replay-сид карт из srs_review_events» читает задвоенную историю → завышенные reps → неверный FSRS-state.

**3. BLOCKER — `word_status` и `study_day` вообще НЕ экспортируются в бандл; план объявляет блокером портируемость лога, но молчит про состояние, ручную ось и стрик-леджер.**
Обоснование: `exportBundle` R-3.7 экспортирует srs_cards/srs_review_events/srs_attempts/srs_card_exports/anki_word_exports/events/translation_overrides (local-db.js:4285–4311) — `word_status` и `study_day` отсутствуют и в export, и в importBundle (grep по файлу: обе таблицы фигурируют только в своём CRUD :2261–2431). После переноса устройства по плану будет лог БЕЗ состояния: (а) ручная ось l1..l4/known/ignore невосстановима из лога вообще; (б) seeded-слова невосстановимы (SM2-колонки-источники исчезли); (в) оракул «replay==state» на мигрированном профиле красный по построению; (г) стрик обнулён при живом review_log. Это ровно инвертированный сценарий мандата — и он реальный, а не гипотетический.
Fix: в P0 добавить в бандл `word_status` (merge: newer-`updated_at`-wins по группам колонок, COALESCE-сохранение source-якорей — паттерн мигр. 060) и `study_day` (merge: per-day MAX(recalls, available)); включить раунд-трип обеих таблиц в `smoke:memory-canon`.

**4. MAJOR — `seedFromSm2` не определён для interval=0 — а это самые горячие строки живого профиля (каждое неверное/skip-ревью даёт interval=0, reps=0); плюс нет ни shadow-mode, ни предсказанного diff перед переключением P2.**
Обоснование: `nextSrs` при ошибке пишет `lapses+1, reps=0, interval=0, due=now` (reader-morph.js:1907–1910) — у активно тренирующегося владельца лапснутые строки в этом состоянии всегда есть. Эвристика §4.3 «S≈interval» даёт S=0 → `retrievability`/`intervalFor` вырождены (деление на S / нулевой интервал навсегда-due). Recon не содержит ни shadow-записи, ни офлайн-прогноза diff на реальном профиле — при том, что владелец пользуется петлёй ежедневно и §2 сам требует measure-before-code.
Fix: специфицировать seed-ветку для interval=0 (эквивалент `initState(Again)` / документированный S-пол) + golden-вектор на неё; перед флипом P2 — офлайн-харнесс: экспорт реального расписания → симуляция N дней под nextSrs и под fsrs-core → отчёт diff размеров очереди/интервалов (дёшево: обе математики pure), либо dual-write FSRS-колонок в P1.5 без смены read-path.

**5. MAJOR — Семантика «после Again» не специфицирована: сегодняшний же-сессионный ре-тест провалов может молча исчезнуть.**
Обоснование: сегодня неверный ответ → `due=now` (reader-morph.js:1910), и `buildTrainSession.isDue` (library-ui.js:1275) возвращает слово в ближайшую же сессию — «провалил → сразу переучил» является частью живого ощущения петли. FSRS Again даёт due из пост-lapse S; если `dueAt` вернёт ≥1 дня, ре-тест провала в тот же день исчезает — фелт-регресс + слабее relearning, и ни один гейт §7 это не ловит (golden-векторы проверяют математику, не продуктовую семантику).
Fix: явный контракт в §4.1: grade 1 → due=now (или короткий same-day шаг short-term памяти FSRS-5/6) + golden-вектор именно на «Again → слово снова due в текущей сессии»; включить в consumers-sweep пункт «isDue после провала».

**6. MAJOR — Skip записывается в канонический лог неотличимо от честного провала (grade 1); политика replay/оптимизатора по `meta.skipped` не задана → либо перманентный D-штраф за «отказался отвечать», либо расхождение оракула.**
Обоснование: §3.2 «skip → grade 1 + meta.skipped=1»; под FSRS grade 1 необратимо повышает difficulty слова (в отличие от сегодняшнего чисто-schedule reset). Пользователь, скипающий name-suspect/опечатку, получает слово с завышенным D → укороченные интервалы навсегда → залипание в ежедневной очереди (LEECH-нудж :1703 смягчает UX, но D в логе уже испорчен). Одновременно §4.1 `replay(logRows)` не говорит, читает ли он meta → если live-путь однажды станет трактовать skip иначе, replay старого лога разойдётся со state (дырка в оракуле), а P6-оптимизатор выучит веса на не-наблюдениях памяти.
Fix: зафиксировать в §4.1/§7: replay применяет skipped-строки РОВНО как live-путь (единая функция), а retention-report/оптимизатор ОБЯЗАНЫ исключать `meta.skipped` (skip — не наблюдение памяти); рассмотреть отдельный код в grade-пространстве лога (напр. grade 0 = no-attempt) вместо перегрузки 1.

**7. MAJOR — P5 «Вспомнил»-тап в чтении: самоотчётное узнавание при ВИДИМОМ слове двигает ручной уровень и FSRS с тем же весом, что тестируемое воспроизведение — инфляция интервалов + тихая смена семантики known-множества кросс-поверхностно.**
Обоснование: §6.2 — «Вспомнил» → grade 3 → review_log + FSRS + `nextLevel(true)`. Сегодня продвижение уровня достижимо только через тестируемый recall со скрытым словом (MC/тайлы/ввод, эскалация по зрелости), а в чтении слово стоит в полном контексте и тестируется лишь направление форма→значение с самооценкой. Одобренная владельцем связка recall→nextLevel (2026-06-28) утверждалась для тестируемого канала. Новый путь позволяет дотапать слова до known без единого воспроизведения; known-состав через `getKnownWordStates` (local-db.js:2249–2257) питает раскраску и i+1 обеих поверхностей → «работающая» семантика статусов деградирует молча.
Fix: reading-tap пишет review_log (source='reading-tap', channel отдельный) и FSRS-состояние, но НЕ двигает ручную ось (или пол-cap, напр. не выше l3); полновесная связка — только тестируемым каналам; пере-решение по данным P6 (recall-rate по source). Это развилка владельца — добавить в §9.

**8. MAJOR — P4 merge-replay снимает существующий local-newer гард и отдаёт порядок фолда чужим часам; плюс replay-писателю нельзя переиспользовать `setWordStatus`.**
Обоснование: сегодня `applyAnkiReviewStates` защищает свежее локальное ревью (`localMs > ankiModMs` → keepLocal, local-db.js:2494–2497), а due Anki вообще mod-производный (`mod + ivl` — index.html:15289–15291). В D3(a) состояние = fold лога по ts, где Anki-ts — часы ДРУГОГО устройства: clock-ahead Anki-ревью встаёт в конец лога навсегда (каждый последующий Room-fold получает отрицательный/нулевой Δt — поведение `nextState` не задано) и пере-двигает due слова, которое владелец только что поревьюил в Зале — ровно то, что старый гард запрещал. Отдельно: писать пересчитанный srs_* через `setWordStatus`(sched) нельзя — он всегда перезаписывает `status=excluded.status` (:2287) и требует статус-аргумент; без COALESCE-дисциплины теряется source-якорь → `getDueReviewCount` (фильтр `srs_surface IS NOT NULL`, :2346) начнёт молча ронять слова из CTA/очереди.
Fix: в §5 специфицировать: (а) clamp future-ts при инжесте (`ts > now+ε` → отклонить/приравнять к now — зеркало future-clamp стрика, reader-morph.js:2005–2006); (б) неотрицательный Δt в `nextState` + golden-вектор на out-of-order; (в) выделенный srs-only UPDATE (не трогает status, COALESCE по якорям) — и внести его в gate-consumers-sweep как второго писателя srs_* (урок multitab v3.11.74–78).

**9. MINOR — Anki-бэкфилл не фильтрует типы ревью: revlog содержит manual-reschedule (type 4, ease 0) и cram — grade 0 не определён в FSRS 1..4.**
Обоснование: `recordAnkiReviews` пишет строки без фильтра по type (local-db.js:2540–2560, `grade: Number(rv.ease) || 0`); §3.2 бэкфиллит `events(source='anki')` в review_log как есть. Существующая retention-метрика уже фильтрует («counts only TRUE review-type reviews», index.html ~15401) — бэкфилл обязан делать то же.
Fix: в спеке бэкфилла — принимать только type∈{0,1,2} (learn/review/relearn) и ease∈1..4; остальное — drop с счётчиком в отчёте идемпотентности.

**10. MINOR — Вырожденная шкала оценок Зала (1|3) будет доминировать в объёме слитого лога; будущий оптимизатор без стратификации исказит Hard/Easy для 4-градационной Студии (один движок, один вес-сет).**
Обоснование: владелец тренируется в Зале, Студия — стаб → лог будет почти бинарным; per-user fit на таком датасете сместит веса, влияющие на все четыре оценки. Плюс каналы (MC-узнавание vs typed-производство vs reading-tap) имеют разную трудность, а FSRS моделирует один след на item. Вред отложен (оптимизатор не сейчас), поля source/channel в схеме уже есть — риск чисто спецификационный.
Fix: в §4.4/§7 зафиксировать: оптимизатор/калибровка стратифицируются по source+channel (и исключают reading-tap/skipped) ДО любого фита; Brier-отчёт P6 — по бакетам source×channel, не пулом.

**11. MINOR — §3.4 обещает неизменную семантику потребителей `srs_*`, но не назначает, КТО проецирует легаси-колонки из FSRS-состояния после переключения.**
Обоснование: `rankByWeakness` читает `_srs.lapses` (reader-morph.js:1940–1946), `getDueWithSource` сортирует `ORDER BY srs_lapses DESC` (local-db.js:2321), leech-нудж читает `sched.lapses` (library-ui.js:1703), `getSrsSchedule` отдаёт interval/reps. FSRS-state содержит reps/lapses, но «interval» не является нативным полем FSRS — если легаси-колонки заморозятся на pre-switch значениях, D4-взвешивание и сортировка очереди тихо деградируют на stale-данных (класс save==paint).
Fix: явный проекционный контракт в §3.4: на каждом ревью fsrs-путь пишет `srs_lapses/reps` из state и `srs_interval := round((due − reviewedAt)/сутки)`; пункт «легаси-проекция обновляется» — в consumers-sweep чеклист P2.
