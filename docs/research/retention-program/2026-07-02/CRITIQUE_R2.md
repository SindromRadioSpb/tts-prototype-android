# R2 (методист SLA) — адверсариальная критика retention-recon (сырой вердикт, 2026-07-02)

{severity: BLOCKER, claim: Recall-first тап (§6.2) фиксирует grade 3 ДО верификации — самоотчёт «Вспомнил» без показа глосса, при видимом слове в подсказывающем контексте, это не retrieval practice, а измерение иллюзии знания, systematically инфлирующее S. Пользователь коммитит «Вспомнил» не увидев ответа — проверить себя невозможно; это хуже канонического flashcard-флоу (Anki: попытка → reveal → оценка). JOL при доступном/выводимом ответе систематически завышены (Koriat & Bjork 2005, «illusion of competence»); вдобавок слово стоит в живом предложении — значение выводимо контекстуальной инференцией без лексического доступа (инференция ≠ retrieval, удержание от неё слабое — Laufer/Hulstijn). Каждый ложный grade 3 идёт в `review_log` и через `replay()` — в S: интервалы растягиваются на неверифицированных данных, провал всплывёт только через месяцы в trainer. fix: перевернуть флоу на reveal-then-grade — тап «Вспомнил?» → попытка про себя → показ глосса → две кнопки «Знал (3) / Не знал (1)»; grade пишется только ПОСЛЕ показа. Стоимость — один экран, дизайн ещё не в коде.}

{severity: MAJOR, claim: «Показать» без записи создаёт MNAR-дыру: провалы систематически недологируются, а §7 «документируем bias» — не митигация, потому что смещённые grade-3 всё равно кормят и replay()-состояние, и P6-калибровку/оптимизатор.
обоснование: Путь успеха = 1 неверифицированный тап (пишет 3); путь провала = 2 тапа с явным самообвинением («Показать» → «Не вспомнил»), причём его можно молча бросить — и это самый вероятный исход именно при провале. Данные missing-not-at-random: лог перепредставляет успехи → observed recall-rate и Brier-калибровка (§7) считаются на выживших, predicted-R «подтверждается» артефактно; оптимизатор (§4.4) фитится на этом же датасете. Документирование смещения в отчёте не убирает его из `review_log`, из которого `replay()` строит боевое состояние.
fix: (а) после «Показать» — обязательная лёгкая пост-оценка «знал/не знал» (совместимо с фиксом BLOCKER: reveal-then-grade закрывает обе дыры одним флоу); (б) локальный счётчик shown-vs-graded в meta → в P6 гейт «reading-tap включается в калибровку только при abandonment < порога»; до тех пор channel='reading-tap' исключён из Brier и из фита весов.}

{severity: MAJOR, claim: Распространение связки recall→nextLevel на reading-тапы (§6.2 «по связке nextLevel(true)») позволяет неверифицированному recognition-самоотчёту дотолкать слово до «known» — т.е. РЕТИРОВАТЬ его из всей системы обучения.
обоснование: Связка одобрена владельцем 2026-06-28 для ТРЕНАЖЁРА, где correct верифицируется машиной (MC против морфо-честных дистракторов, typed против accepted-skeletons — `checkTrainAnswer` library-ui:1678). Reading-тап — самоотчёт (см. BLOCKER), но двигает ту же лестницу тем же +1 (`nextLevel` reader-morph:1894, `_LV_UP` до known). Последствия «known» системные: слово уходит из i+1-frontier (`corpus-vocab.KNOWN_STATES`), из регулярных сессий (только capped-освежение ~15%, `buildTrainSession` :1281), стена красится «чистым» — при этом слово ни разу не прошло ни одной верифицированной проверки, тем более production-тира (эскалация known=ввод так и не наступит: слово уже «известно»). Пять тапов «Вспомнил» на удобных контекстах = слово исчезло из обучения.
fix: reading-тап пишет ТОЛЬКО grade в `review_log` (двигает расписание); `nextLevel` двигается только верифицированными ответами тренажёра. Компромисс-вариант, если владелец хочет сохранить связку: reading-driven промоушен капится на l3 (входной порог production-тира), последний шаг l4→known — только через typed/tiles.}

{severity: MAJOR, claim: Одна S на слово при четырёх каналах разной глубины (mc-recognition / tiles / typed-production / reading-self-report / anki) противоречит transfer-appropriate processing, а Anki-merge «оба считаются» конкретно голодоморит очередь Зала для активно изучаемых слов.
обоснование: Рецептивное и продуктивное знание слова — разные конструкты с разной динамикой забывания (Nation 2001; Webb 2005); выгода retrieval practice максимальна для ПРАКТИКУЕМОГО маршрута извлечения (Morris/Bransford/Franks 1977). FSRS моделирует одну однородную память; `replay()` фолдит все каналы → S = «средняя знакомость», predicted-R не интерпретируем ни для одного канала, оптимизатор учит усреднённый decay. Конкретный сценарий §5+§3.3: fan-in «много Anki-карт → один канон-ключ» — пользователь ежедневно ревьюит 2–3 recognition-карты одной леммы → серия grade≥3 → S взлетает → слово уходит из due Зала → reading-native retrieval и production-тир НЕ практикуются именно на словах, которые пользователь активно учит (моат каннибализирован компаньоном). Бонус-рассинхрон: Anki двигает S/due, но не level → слово «зрелое» по расписанию визит раз в полгода, но вечное l1/MC и вечное «В работе» в бейдже (R11 кросс-поверхностная несогласованность зрелости).
fix: минимум — аттенюация не-Зал-каналов при fold (anki Good → эквивалент Hard, документированно) + запрет каналам recognition-класса двигать level; честный вариант — двух-трековое состояние: production-S из channel∈{typed,tiles,dictate} + reverse, recognition-S из остальных (`replay(log.filter(...))` — движок уже pure, фильтр бесплатный), due Зала от той оси, которую Зал реально тренирует. Это развилка уровня D1–D7 — добавить в §9 на решение владельца, сейчас её там нет.}

{severity: MAJOR, claim: FSRS будет откалиброван на вырожденном cue: и open-text сессия, и cross-text due-очередь предъявляют слову ОДИН И ТОТ ЖЕ якорный контекст на каждом ревью — S начинает мерить память предложения, а не знание слова.
обоснование: Выбор occurrence детерминирован («предложение с наибольшим числом токенов», 3b-док §дефолты), D2 хранит единственный source-якорь (`srs_text_key/sentence_id/order_index`, мигр. 060) и `getSentenceForReview` ре-анкорит в него же — слово, поревьюенное 8 раз, видит идентичный клоуз 8 раз. Энкодинг-специфичность (Tulving & Thomson 1973): успех извлечения привязывается к конкретному cue; повторный идентичный контекст → item-меморизация («слово из ЭТОГО предложения»), успех перестаёт индицировать лексическое знание, вариативность контекста нужна для переноса (Smith/Handy). Рекон строит FSRS-состояние и §7-калибровку поверх этого сигнала — интервалы растянутся на предложение-знакомости; recall в новом контексте (собственно чтение) будет систематически ниже predicted-R.
fix: ротация якоря — в open-text выбирать occurrence round-robin по reps (`occ` уже собран `_scanWords`); для cross-text хранить 2–3 якоря или lastServedSentenceId с предпочтением другого предложения того же текста (order_index-скан). Дёшево, ложится в P0/P2.}

{severity: MINOR, claim: Пост-teach немедленный MC становится ПЕРВОЙ строкой `review_log` слова (grade 3 при потолочной вероятности успеха) — S0 теряет диагностическую ценность, оптимизатор получает шум.
обоснование: D5-флоу (library-ui:1522): teach-показ (не пишет — верно) → тут же счётный MC того же слова; ответ показан секунды назад → успех из рабочей памяти, не из долговременной (Karpicke & Roediger 2008: немедленный тест ≈ restudy). `initState(3)` для такого события формально совпадает с Anki-first-review, но MC-4 после показа даёт ~100% pass независимо от качества кодирования → распределение первых оценок вырождено в 3, S0 неинформативен.
fix: писать `meta.postTeach=1`; в P6 исключать такие строки из фита весов/Brier; опционально трактовать в replay как learning-step (short-lag), а не полноценное ревью.}
