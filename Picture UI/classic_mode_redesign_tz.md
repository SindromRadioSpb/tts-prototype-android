# ТЗ на редизайн UI/UX режима Classic Mode
## Проект: text-to-speech / translation / table workflow
## Платформы: Desktop Web + Mobile Web
## Уровень качества: Premium / Production-grade

---

## 1. Контекст и основание для задачи

Редизайн требуется на основании анализа текущих интерфейсов **Classic Mode** в desktop- и mobile-версиях по приложенным скриншотам.

Исходное наблюдение: текущий экран функционально насыщен, но воспринимается как **техническая панель с большим количеством разрозненных контролов**, а не как профессиональный рабочий продукт премиального уровня.

Особенности входных материалов:
- desktop-скриншот отражает текущую компоновку полного рабочего экрана;
- mobile-скриншот представляет собой **сшивку нескольких экранов при скроллинге**;
- повторяющаяся кнопка **IDE Mode** на mobile-скриншоте не трактуется как несколько отдельных экземпляров UI-компонента, а считается следствием сшивки.

---

# 2. Цель редизайна

## 2.1. Главная цель

Преобразовать Classic Mode из перегруженного набора элементов управления в **премиальный guided workspace**, где пользователь быстро и уверенно проходит основной сценарий:

**ввод текста → настройка обработки → озвучка / перевод → просмотр результата → экспорт / дальнейшие действия**

## 2.2. Продуктовый результат

После редизайна интерфейс должен:
- мгновенно объяснять пользователю главный сценарий;
- иметь сильную визуальную и функциональную иерархию;
- одинаково уверенно работать на desktop и mobile;
- поддерживать сложный мультиязычный контент без визуального хаоса;
- выглядеть как зрелый продукт, а не как internal utility screen.

---

# 3. Основные проблемы текущего интерфейса

## 3.1. Проблемы информационной архитектуры
1. Смешаны в одной плоскости:
   - ввод текста,
   - системные квоты,
   - выбор TTS,
   - перевод,
   - навигация по приложению,
   - экспорт,
   - настройки таблицы,
   - статусные сообщения.
2. Глобальная навигация находится внутри основного рабочего потока.
3. Нет ясного деления на этапы сценария.
4. Нет явного различия между primary, secondary и service actions.

## 3.2. Проблемы визуальной иерархии
1. Слишком много элементов одного визуального веса.
2. Второстепенные блоки конкурируют с главным действием.
3. Disabled-состояния выглядят как “сломанный UI”, а не как контролируемая недоступность.
4. Микротексты и служебные подписи создают шум.

## 3.3. Проблемы mobile UX
1. Сценарий растянут по длинной вертикали.
2. Слишком много селектов, кнопок и служебных плашек до результата.
3. Важные действия и второстепенные перемешаны.
4. Floating-контролы перекрывают контент.
5. Таблица выглядит как сжатая desktop-версия, а не как mobile-first решение.

## 3.4. Проблемы доверия и продуктового восприятия
1. Пользователь не всегда мгновенно понимает:
   - актуален ли результат,
   - нужно ли пересоздание,
   - что именно устарело,
   - почему какие-то действия недоступны.
2. Смешение языков интерфейса снижает ощущение целостности.
3. Экран создает впечатление “всё доступно сразу”, но не ведет пользователя через сценарий.

---

# 4. Область редизайна

Редизайн охватывает **Classic Mode** целиком, включая:

1. Верхнюю область экрана:
   - usage/квоты/статусы провайдеров;
2. Блок ввода текста;
3. Блок выбора языка и настроек отображения;
4. Блок TTS:
   - provider,
   - voice,
   - speed,
   - pitch,
   - action;
5. Блок AI translation / transliteration;
6. Локальные action buttons;
7. Навигационные элементы, если они присутствуют на экране Classic Mode;
8. Result summary / provenance / draft status;
9. Toolbar таблицы;
10. Саму таблицу;
11. Export / downstream actions;
12. Поведение на mobile;
13. Систему состояний и обратной связи.

---

# 5. Что не является задачей этого этапа

На данном этапе **не является обязательным**:
- смена core-бизнес-логики провайдеров;
- изменение внутренней логики TTS / translation pipeline;
- замена табличной модели данных;
- удаление Classic Mode как режима;
- глубокий backend-refactor, если он не нужен для реализации UX.

Однако допускаются технические изменения, если без них невозможно реализовать качественный UI/UX и корректную state machine.

---

# 6. Целевая UX-модель

## 6.1. Базовая концепция

Classic Mode должен стать **сценарным рабочим пространством**, а не flat-списком контролов.

Пользователь должен видеть:
1. **Что он сейчас редактирует**
2. **Что с этими данными можно сделать сейчас**
3. **Какое главное действие на текущем этапе**
4. **Что уже построено**
5. **Какие дальнейшие действия доступны**

## 6.2. UX-принципы редизайна

1. **One primary action at a time**
   На каждом шаге должен доминировать один основной action.

2. **Progressive disclosure**
   Второстепенные и редкие настройки не должны перегружать основной экран.

3. **State clarity**
   Статус результата, актуальность данных и необходимость пересборки должны считываться мгновенно.

4. **Cross-platform coherence**
   Desktop и mobile должны быть логически едины, но по компоновке — адаптированы под устройство.

5. **Premium calmness**
   Интерфейс должен быть визуально спокойным, без ощущения “панели оператора”.

6. **Multilingual robustness**
   Интерфейс должен уверенно работать с RTL/LTR/кириллицей/латиницей/транслитом.

# 7. Целевые пользовательские сценарии

## 7.1. Основной сценарий
1. Пользователь вставляет или вводит текст.
2. Выбирает или подтверждает язык.
3. Настраивает TTS/voice при необходимости.
4. Настраивает translation/translit при необходимости.
5. Нажимает главное действие.
6. Получает таблицу и аудио/промежуточный результат.
7. Выполняет downstream action:
   - прослушивание,
   - работа по строкам,
   - экспорт,
   - сохранение.

## 7.2. Повторный сценарий
1. Пользователь меняет исходный текст.
2. Интерфейс ясно показывает, что текущие результаты устарели.
3. Пользователь выполняет обновление.
4. Интерфейс показывает новый актуальный результат.

## 7.3. Сценарий работы с уже существующим результатом
1. Пользователь открывает экран с уже восстановленным результатом.
2. Видит, что данные загружены из локального состояния/кэша.
3. Понимает, насколько результат актуален относительно текущего текста.
4. Может либо использовать существующий результат, либо пересобрать его.

## 7.4. Mobile quick-use сценарий
1. Пользователь открывает экран на телефоне.
2. Сразу видит текст, основное действие и краткий статус.
3. Не обязан проходить через длинный набор второстепенных контролов до результата.
4. Может быстро запустить обработку и увидеть результат в мобильном виде.

# 8. Требования к новой информационной архитектуре

## 8.1. Экран должен быть разделён на 5 логических слоёв

### Слой 1. Header / global status
Содержит:
- usage / quota summary;
- provider/system status;
- при необходимости — глобальную навигацию.

Не должен конкурировать с рабочим сценарием.

### Слой 2. Input composer
Содержит:
- текст для обработки;
- статус текста;
- language context;
- быстрые действия по input.

### Слой 3. Processing controls
Содержит:
- TTS settings;
- translation settings;
- primary actions.

### Слой 4. Result surface
Содержит:
- result header;
- provenance;
- актуальность;
- summary;
- table toolbar.

### Слой 5. Output / export actions
Содержит:
- audio;
- anki;
- table export;
- playlist / sequence actions.

---

## 8.2. Глобальная навигация должна быть вынесена из workflow-линии

Кнопки вида:
- Библиотека
- Дашборд
- SRS Trainer
- Обновить

не должны разрывать рабочий процесс внутри центральной зоны обработки текста.

### Требование
Нужно явно разделить:
- **global navigation**
- **screen-local actions**
- **result/export actions**

### Допустимые варианты
- top app bar;
- sticky segmented navigation;
- compact nav strip;
- bottom nav на mobile.

---

# 9. Desktop UX requirements

## 9.1. Компоновка desktop

Desktop-версия должна перейти от “одной длинной вертикальной формы” к зрелой рабочей компоновке.

### Предпочтительный вариант
**Двухзонная layout-модель**:

#### Левая/основная зона
- input composer;
- TTS controls;
- translation controls;
- primary actions.

#### Правая/результатная зона
- result summary;
- draft/current/stale status;
- provenance;
- export actions;
- table preview/full table.

### Альтернативно допустимо
Сверху:
- input + controls
Снизу:
- result area full width

Но в любом случае экран должен иметь **явное разделение “настройка/действие” и “результат”**.

---

## 9.2. Usage и quota должны стать компактным статусным слоем

Верхние карточки usage не должны занимать непропорционально много внимания.

### Требования
1. Уменьшить визуальную доминанту usage cards.
2. Свести их в compact system status strip.
3. При необходимости — раскрывать подробности по click/tap.
4. Google Cloud Console — вторичный action, не конкурирующий с рабочим сценарием.

---

## 9.3. Input composer должен быть полноценным рабочим блоком

### Обязательно
- явный заголовок/label;
- placeholder / example / usage hint;
- состояние empty / filled / dirty / processing-related;
- быстрые действия: clear / paste / restore / expand при необходимости;
- чёткая визуальная роль “здесь начинается сценарий”.

### Недопустимо
Textarea не должен выглядеть как просто большая пустая коробка между системными блоками.

---

## 9.4. Processing controls должны быть сгруппированы по смыслу

Нужно отделить:
- настройки TTS;
- настройки translation/translit;
- локальные действия.

### Требование
Не смешивать в одной линии:
- провайдер озвучки,
- провайдер перевода,
- глобальную навигацию,
- экспорт.

---

## 9.5. Result area должна иметь сильную шапку

Над таблицей должен быть компактный и информативный result header, содержащий:
- статус результата: актуален / устарел / черновик / восстановлен;
- источник;
- provider summary;
- niqqud/translit status;
- last updated;
- availability downstream actions.

---

# 10. Mobile UX requirements

## 10.1. Mobile должен проектироваться как отдельный сценарий, а не как ужатый desktop

Основная ошибка текущего состояния — mobile-поведение ощущается как длинная вертикальная версия desktop-form.

### Требование
Mobile должен быть redesign-first, а не desktop-shrink.

---

## 10.2. Mobile first-screen must contain the essentials

На первом экране после входа пользователь должен увидеть:
- input;
- status;
- primary action;
- только самые нужные controls.

### Недопустимо
Сначала длинный набор второстепенных селектов и навигационных кнопок, а затем основной action.

---

## 10.3. Mobile workflow должен строиться по step-like схеме

Рекомендуемая последовательность:
1. Text input
2. Main action
3. Key settings
4. Result
5. Advanced settings / export

---

## 10.4. Второстепенные настройки должны быть свернуты

На mobile нужно вынести в:
- accordion;
- modal sheet;
- bottom sheet;
- expandable “Дополнительно”.

### Сюда относятся
- шрифт табличной части;
- расширенные режимы отображения таблицы;
- вторичные служебные параметры;
- часть интеграционных действий.

---

## 10.5. Floating utility controls не должны перекрывать контент

### Обязательное требование
Плавающие кнопки не должны:
- закрывать текст;
- перекрывать строки таблицы;
- лежать поверх CTA;
- ломать чтение RTL/LTR-контента.

### Для IDE Mode
Требуется переосмыслить размещение:
- либо в app bar;
- либо в utility zone;
- либо в скрывающийся FAB, не перекрывающий контент;
- либо как fixed switch в безопасной зоне.

---

## 10.6. Таблица на mobile должна иметь собственный mobile pattern

### Обязательно
- sticky header;
- controlled horizontal scroll только внутри table container;
- крупные tap targets;
- безопасная работа с mixed text directions;
- mobile row inspection;
- возможность раскрыть строку в bottom sheet / detail card.

### Недопустимо
Просто уменьшенная desktop-table без mobile-паттернов.

---

# 11. Требования к action hierarchy

## 11.1. Все действия должны быть разделены на уровни

### Primary
Главное действие текущего шага:
- Перевести и сохранить
- Озвучить
(в зависимости от сценария и приоритета этапа)

### Secondary
Важные вспомогательные действия:
- Обновить
- Изменить режим
- Повторить
- Настроить

### Tertiary / utility
- открыть дополнительные настройки;
- раскрыть метаданные;
- служебные действия.

### Disabled
Действия должны быть не просто серыми, а **контекстно объяснёнными**.

---

## 11.2. На одном экране не должно быть нескольких конкурирующих primary actions без ясной логики

Если на экране есть и “Озвучить”, и “Перевести и сохранить”, то:
- либо должен быть доминирующий главный сценарий;
- либо должен быть segmented mode/choice;
- либо actions должны быть разведены по сценарным секциям.

---

## 11.3. Недоступные действия должны быть умными

### Допустимое поведение
- скрывать до релевантного состояния;
- показывать compact disabled state;
- объяснять причину недоступности.

### Недопустимо
Длинные ряды неактивных серых кнопок без пользы и без объяснения.

---

# 12. Требования к работе со статусами и состояниями

## 12.1. Должна быть единая state model для всего экрана

Минимальный набор состояний:

### Input state
- empty
- ready
- dirty

### Processing state
- idle
- in progress
- succeeded
- partial
- failed

### Result state
- absent
- draft
- ready
- stale
- restored from cache
- outdated by source edit

### Export state
- unavailable
- available
- in progress
- completed
- failed

---

## 12.2. Статус актуальности результата должен быть очевиден

Пользователь всегда должен мгновенно понимать:
- соответствует ли таблица текущему тексту;
- соответствует ли аудио текущему тексту;
- требует ли translit/translation пересборки;
- откуда взят текущий результат.

---

## 12.3. Warning/notice/confirmation должны быть системными

Жёлтые уведомления и информационные плашки должны стать частью единой системы:
- info
- warning
- success
- error

Нужно убрать ощущение случайных разнотипных сообщений.

---

# 13. Требования к таблице результата

## 13.1. Над таблицей нужен профессиональный toolbar

Toolbar должен содержать:
- режимы отображения;
- presets колонок;
- playlist / sequence controls;
- optional row density;
- настройку колонок;
- reset view.

## 13.2. Управление колонками должно стать двухуровневым

### Уровень 1
Быстрые пресеты:
- Полная
- Иврит + рус
- Фонетика
- Только иврит

### Уровень 2
Тонкая настройка колонок в popover / drawer / bottom sheet.

### Недопустимо
Постоянно держать все чекбоксы и все режимы открытыми на mobile и перегружать desktop.

---

## 13.3. Row actions должны быть приведены к единому стандарту

### Обязательно
- единый размер hit area;
- понятные иконки;
- hover / focus / pressed states;
- tooltips на desktop;
- доступность на touch.

---

## 13.4. Metadata блока результата должна быть компактной и сильной

Источник, provider, provenance, status — важны, но должны быть оформлены как:
- compact info chips;
- metadata row;
- summary line.

Они не должны визуально спорить с таблицей.

---

# 14. Требования к текстам интерфейса и локализации

## 14.1. Нужна единая языковая политика интерфейса

Недопустимо бессистемное смешение:
- русского,
- английского,
- технических терминов без локализационной логики.

### Требование
Определить и реализовать один из режимов:
1. Полностью русский UI
2. Билингвальный UI по правилам
3. Выбор language pack

Для premium-версии предпочтительно: **единый русский UI**, если экран ориентирован на русскоязычную аудиторию.

---

## 14.2. Терминология должна быть унифицирована

Требуется product vocabulary:
- Перевести и сохранить
- Озвучить
- Настройки озвучки
- Настройки перевода
- Таблица
- Источник
- Черновик
- Актуально
- Требует обновления
- Восстановлено локально
- Недоступно

---

## 14.3. Микрокопирайтинг должен быть коротким и уверенным

Нужны:
- краткие labels;
- объяснимые ошибки;
- ясные disabled reasons;
- понятные статусы обработки;
- простые подтверждения успеха.

---

# 15. Требования к design system

## 15.1. Единая визуальная система компонентов

Необходимо унифицировать:
- button height;
- input/select heights;
- radius;
- spacing scale;
- border style;
- focus styles;
- chips;
- cards;
- alerts.

## 15.2. Типографическая шкала

Обязательна иерархия:
- H1 — screen title
- H2 — section title
- Body
- Caption
- Meta

Не должно быть ситуации, когда служебный текст и функциональный заголовок читаются как элементы одного уровня.

## 15.3. Цветовая система

Должна быть единая система для:
- primary action;
- secondary action;
- success;
- warning;
- error;
- info;
- disabled.

### Требование
Disabled не должен выглядеть как broken UI.

---

# 16. Accessibility requirements

## 16.1. Touch / target size
Все интерактивные элементы на mobile:
- не менее 44x44 px;
- достаточный отступ между соседними controls.

## 16.2. Contrast
Проверить контраст:
- мелких подписей;
- grey disabled labels;
- warning banners;
- info chips;
- link states.

## 16.3. Keyboard accessibility on desktop
Обязательно:
- корректный tab order;
- видимый focus;
- предсказуемая работа клавиатуры;
- без focus-traps;
- горячие клавиши не должны создавать визуальный шум.

## 16.4. RTL/LTR robustness
Для mixed Hebrew/Russian/Latin/SBL text обязательно:
- direction-safe layout;
- корректная bidi isolation;
- устойчивые table cells;
- предсказуемое выравнивание по языковому контексту.

# 17. Motion and interaction polish

## 17.1. Микроанимации
Допустимы только если помогают:
- раскрытие advanced settings;
- переход в loading;
- появление success/error states;
- смена status chips.

Не допускается декоративная анимация ради анимации.

## 17.2. Scroll behavior
Нужно исключить:
- случайные overlay conflicts;
- прыжки layout;
- перескоки фокуса;
- перекрытие fixed/floating controls.

# 18. Технические UI/UX constraints

## 18.1. Нельзя использовать фиксированные высоты там, где контент может расти
Это особенно критично для:
- input area;
- alerts;
- result summary;
- table toolbars;
- mobile sheets;
- multilingual content.

## 18.2. Длинный контент должен иметь корректную прокрутку
Прокрутка должна быть:
- предсказуемой;
- локализованной в нужном контейнере;
- без двойной конкурирующей прокрутки, где это не нужно.

## 18.3. Безопасные отступы и inset support
Особенно для mobile:
- safe areas;
- fixed bars;
- floating controls;
- bottom actions.

## 18.4. UI не должен ломаться на узких ширинах
Нужно проектировать минимум для:
- узких мобильных экранов;
- длинных строк на иврите;
- длинных русских переводов;
- mixed-content rows.

# 19. Data/status transparency requirements

## 19.1. Provenance и source-of-truth должны быть читаемы
Пользователь должен понимать:
- какой provider использовался;
- есть ли niqqud;
- есть ли translit;
- был ли результат восстановлен из local cache;
- актуален ли он.

## 19.2. Квоты и usage должны быть понятны, но не шумны
Показывать:
- usage summary;
- thresholds;
- contextual warnings;
- links to quota management как secondary path.

# 22. Definition of Done

Редизайн считается завершённым только если выполнены все условия ниже.

## 22.1. Desktop DoD
1. Пользователь видит главный workflow за 3–5 секунд.
2. Input, processing, result и export визуально и логически разделены.
3. Usage/status не конкурирует с основным сценарием.
4. Таблица имеет зрелую result header + toolbar model.
5. Глобальная навигация вынесена из основного потока.

## 22.2. Mobile DoD
1. На первом экране присутствуют input + status + primary action.
2. Нет необходимости длинного скролла до основного действия.
3. Floating-controls не перекрывают контент.
4. Secondary settings свёрнуты и не перегружают экран.
5. Таблица пригодна для реального mobile-использования.

## 22.3. Common UX DoD
1. Нет хаотичного смешения русского и английского UI.
2. Все состояния экрана читаются мгновенно.
3. Недоступные действия объясняются или скрываются.
4. Visual hierarchy очевидна.
5. Multilingual content отображается устойчиво.
6. Accessibility и responsive-поведение проверены.

# 24. Риски редизайна

## 24.1. Риск №1 — косметический редизайн без перестройки IA
Если изменить только стили, а не архитектуру экрана, продукт не перейдёт на premium-уровень.

## 24.2. Риск №2 — desktop-first без mobile rethink
Просто адаптировать desktop на mobile недостаточно; нужен отдельный mobile UX.

## 24.3. Риск №3 — попытка оставить все controls видимыми
Premium достигается не количеством показанных функций, а качеством их раскрытия.

## 24.4. Риск №4 — отсутствие единой state model
Без этого экран останется неоднозначным по состояниям и доверию.

## 24.5. Риск №5 — смешение продуктовой и технической лексики
Это сохраняет впечатление внутреннего utilitarian tool, а не зрелого продукта.
