# classic_mode_redesign_tz_v2

## Проект
`tts-prototype-android`

## Документ
Product / UX source-of-truth для второй волны редизайна `Classic Mode`.

## Статус
`v2`

## Уровень качества
**Ultra-premium / executive-grade / trust-heavy product surface**

## Целевой стандарт
Интерфейс должен соответствовать ожиданиям от продукта с подпиской **100 000 ₪ в месяц**:  
не просто “работать удобно”, а демонстрировать **безошибочную ясность, дорогой визуальный язык, предсказуемое поведение, высокое доверие, сильную мобильную пригодность и зрелую операционную дисциплину**.

---

# 1. Контекст v2

Первая волна редизайна уже решила значимую часть structural UX debt:

- экран перестал быть хаотичной технической панелью;
- появился более ясный guided workflow;
- desktop стал ближе к workspace-модели;
- mobile стал заметно лучше;
- появились более сильные статусы;
- mobile row-detail sheet стал качественным шагом вперёд.

Однако после первой итерации состояние экрана всё ещё нельзя считать premium-complete.

`v2` требуется не для косметической доработки, а для **целенаправленного premium hardening**.

---

# 2. Цель v2

## 2.1. Главная цель

Довести `Classic Mode` до состояния, в котором интерфейс:

- выглядит дорого и уверенно;
- мгновенно объясняет главный сценарий;
- не содержит визуальных компромиссов и артефактов;
- одинаково убедителен на desktop и mobile;
- вызывает доверие как рабочая поверхность для текста, озвучки, перевода и таблицы;
- воспринимается как часть high-end образовательного продукта, а не как сложный utility screen.

---

## 2.2. Практический результат

После `v2` пользователь должен ощущать, что:

1. интерфейс знает, что сейчас главное;
2. система ясно показывает актуальность и происхождение результата;
3. mobile не является компромиссной версией desktop;
4. никакие утилитарные элементы не мешают работе;
5. весь экран имеет спокойный, дорогой, управляемый ритм;
6. сложные возможности раскрываются по требованию, а не кричат с первого экрана.

---

# 3. Продуктовая планка для уровня 100 000 ₪ / month

Для продукта такого класса недостаточно:
- “стало лучше”;
- “стало понятнее”;
- “всё работает”.

Необходимо добиться одновременно пяти свойств:

## 3.1. Безусловная ясность
Пользователь за 3–5 секунд понимает:
- где вход;
- какой текущий статус;
- какое главное действие;
- что уже готово;
- что можно делать дальше.

## 3.2. Операционная надёжность
UI не вводит в заблуждение относительно:
- stale/current состояния;
- сохранённости;
- происхождения результата;
- готовности к экспорту;
- доступности downstream actions.

## 3.3. Визуальная дороговизна
Экран должен транслировать:
- спокойствие;
- контроль;
- премиальную собранность;
- отсутствие случайности;
- отсутствие “сборки из кусков”.

## 3.4. Mobile-first зрелость
Телефонная версия должна быть не “терпимой”, а действительно сильной, с:
- быстрым first-screen flow;
- guided behavior;
- минимальным шумом;
- хорошей touch ergonomics;
- detail-first моделью для сложных данных.

## 3.5. Trust-heavy UX
Система должна выглядеть так, будто ей можно доверить серьёзный учебный и рабочий процесс:
- без странных overlay-эффектов;
- без двусмысленных действий;
- без неясных disabled-состояний;
- без ощущения “недополированного инструмента”.

---

# 4. Базовый post-v1 baseline

## 4.1. Что уже считается достигнутым и не должно быть утрачено

### Сохранить обязательно:
1. разделение input / actions / result / table;
2. улучшенную desktop workspace-логику;
3. улучшенный mobile first-screen compared to old version;
4. state chips / warning/status language как основу trust layer;
5. row-detail bottom sheet на mobile как удачный паттерн;
6. общее движение от хаотичного control wall к guided structure.

## 4.2. Что считается unresolved после v1
1. экран ещё не выглядит бесспорно ultra-premium;
2. primary action hierarchy ещё не везде железобетонна;
3. mobile всё ещё можно существенно сократить и успокоить;
4. floating utility/FAB behaviour остаётся потенциальным premium-defect;
5. table control layer на mobile всё ещё тяжеловат;
6. result surface ещё можно сделать сильнее;
7. top status/usage layer ещё слишком массивен;
8. visual rhythm/design language не доведены до финальной дорогой зрелости.

---

# 5. Главные задачи v2

## 5.1. Premium hardening
Довести экран до визуально дорогого состояния:
- спокойный ритм;
- дорогая композиция;
- меньший шум;
- более зрелая типографика;
- меньшая конкуренция вторичных элементов;
- меньше ощущения “операторского экрана”.

## 5.2. Action hierarchy hardening
Сделать так, чтобы:
- primary action был однозначен;
- stale-state менял поведенческий приоритет экрана;
- export/downstream actions не спорили с текущим шагом;
- navigation не отвлекала от workflow.

## 5.3. Mobile hardening
Сделать mobile:
- короче;
- чище;
- guided-first;
- detail-first для строк;
- менее table-mechanics-heavy;
- более похожим на high-end app screen, а не на адаптированный web admin.

## 5.4. Trust layer hardening
Усилить:
- provenance;
- актуальность;
- сохранённость;
- export readiness;
- restored/local state messaging;
- ясность disabled/hidden logic.

6. translation + transliteration settings;

13. responsive behavior;
14. visual polish and design rhythm.

# 7. Что не является задачей v2

Если не требуется для UX-результата, в `v2` не обязательно:
- менять core business logic провайдеров;
- менять pipeline перевода/озвучки по сути;
- переписывать data model таблицы;
- удалять Classic Mode как режим;
- делать полный архитектурный rewrite всего фронтенда.

Однако допускаются технические изменения, если они необходимы для:
- качественного premium UX;
- стабильной state model;
- устранения overlay/interaction конфликтов;
- повышения testability и regression safety.

---

# 8. Целевая UX-модель v2

## 8.1. Новая формула экрана
Classic Mode должен восприниматься как:

**guided learning/processing workspace**,  
а не как:
- набор контролов,
- экран инженера,
- legacy utility form.

## 8.2. Пользователь всегда должен мгновенно понимать
1. Что является входом.
2. Какой статус у текущего содержимого.
3. Какое действие сейчас главное.
4. Что уже собрано и насколько это актуально.
5. Что делать после получения результата.

---

# 9. Desktop premium expectations

## 9.1. Desktop должен ощущаться как executive workspace
Desktop-экран должен транслировать:
- сильную композицию;
- чёткое разделение рабочей и результатной областей;
- спокойную плотность;
- уверенную структуру действий.

## 9.2. На desktop не должно быть:
- визуальной конкуренции двух равносильных CTA без причины;
- ощущения перегруженности служебными вещами;
- “формы с множеством полей” вместо product workspace;
- тяжёлой usage-панели, которая спорит с главным экраном.

## 9.3. Desktop должен получить во v2
1. более жёсткую dominant CTA hierarchy;
2. более дорогой result header;
3. более компактный и спокойный top status strip;
4. более зрелую настройку cards/settings surfaces;
5. более polished data table impression.

---

# 10. Mobile premium expectations

## 10.1. Mobile не должен ощущаться как compressed desktop
Даже если используется одна кодовая база, мобильное поведение должно быть логически самостоятельным.

## 10.2. Первый экран mobile обязан быть ruthless по приоритетам
На первом экране пользователь должен увидеть:
- input;
- статус;
- главное действие;
- минимум отвлекающих вещей.

## 10.3. Mobile должен быть detail-first
Для сложных строк и таблицы основным паттерном должен стать:
- быстрый обзор;
- тап по строке;
- detail sheet / card interaction.

Не допускается reliance на “узкую desktop-table в телефоне” как на основной UX.

## 10.4. Mobile во v2 должен получить
1. ещё более короткий first-screen contract;
2. более глубокий progressive disclosure;
3. упрощённый table control layer;
4. окончательно решённую судьбу floating utility control;
5. усиленный row-detail sheet;
6. более спокойный result/export layer.

---

# 11. Основные премиальные дефекты, которые v2 обязан устранить

## 11.1. Overlay / utility defects
Любой floating utility control, который:
- перекрывает контент;
- конкурирует с CTA;
- мешает чтению;
- выглядит как overlay-артефакт,

должен быть устранён как premium-defect.

## 11.2. Action ambiguity
Если экран знает, что результат устарел, интерфейс не имеет права визуально спорить сам с собой относительно главного следующего шага.

## 11.3. Mobile control heaviness
Если mobile показывает слишком много низкоуровневых table mechanics на поверхности, это считается незавершённой работой.

## 11.4. Result fragmentation
Если результат всё ещё ощущается как набор подблоков, а не как единая сильная surface, это нужно исправить.

## 11.5. Visual semi-premium state
“Уже лучше и уже хорошо” не является достаточным результатом для `v2`.

---

# 12. Требования к информационной архитектуре v2

## 12.1. Экран должен состоять из пяти логических зон

### 1. System status / global utility
Компактный, тихий, вторичный слой.

### 2. Input and immediate workflow
Исходный текст, локальные статусы, primary action logic.

### 3. Processing settings
TTS + translation/translit controls, но без form-like перегруза.

### 4. Result surface
Сильная единая область с result header, provenance, export readiness и table entry.

### 5. Detail and downstream actions
Таблица, mobile detail sheets, export, row-level interactions.

---

## 12.2. Navigation должна быть вторичной к сценарию
Global navigation не должна нарушать ощущение guided flow.

---

# 13. Требования к action hierarchy v2

## 13.1. Один главный шаг за раз
Primary action должен быть:
- самым очевидным;
- самым сильным;
- логически оправданным текущим состоянием.

## 13.2. Secondary actions не спорят с primary
Они могут быть рядом, но не должны эмоционально или визуально перетягивать внимание.

## 13.3. Export actions контекстны
Экспорт/Audio/Anki/Download не должны:
- висеть как “мертвые серые плиты”;
- конкурировать с текущей задачей;
- появляться сильнее, чем сам рабочий сценарий.

---

# 14. Требования к state model v2

## 14.1. Состояния должны быть предельно читаемы
Минимум:
- empty;
- ready;
- dirty;
- processing;
- ready/current;
- restored;
- stale;
- error;
- export-ready / export-unavailable.

## 14.2. Экран должен безошибочно показывать:
- что устарело;
- что актуально;
- что сохранено;
- что восстановлено;
- что недоступно и почему.

## 14.3. Визуальные состояния должны быть частью дорогой product language
Статусы — не просто техтеги, а часть премиального доверительного слоя.

---

# 15. Требования к result surface v2

## 15.1. Result area должна стать одной из сильнейших зон экрана
Нужно усилить:
- result header;
- provenance;
- export readiness;
- status coherence;
- логическую цельность result surface.

## 15.2. Пользователь должен мгновенно понимать:
- откуда результат;
- актуален ли он;
- сохранён ли он;
- можно ли уже экспортировать;
- что рекомендуется сделать следующим шагом.

---

# 16. Требования к table UX v2

## 16.1. Table на desktop
Должна оставаться data-rich, но стать:
- визуально легче;
- более дорогой;
- более disciplined;
- с cleaner row actions;
- с better toolbar composition.

## 16.2. Table на mobile
Должна перестать быть главной тяжёлой surface.  
Основной mobile pattern:
- preset view;
- short overview;
- row tap;
- detail sheet interaction.

## 16.3. Mobile table settings
На поверхности должны остаться только наиболее ценные и частые сценарии.  
Тонкие чекбоксные механики — глубже.

---

# 17. Требования к row-detail sheet v2

## 17.1. Row-detail sheet — стратегически правильное направление
Этот паттерн нужно сохранить и усилить.

## 17.2. Sheet должен стать ещё дороже
Нужно улучшить:
- иерархию секций;
- типографику;
- CTA semantics;
- баланс secondary actions;
- визуальную чистоту;
- отсутствие конфликтов с floating utility controls.

---

# 18. Требования к visual language v2

## 18.1. Экран должен выглядеть дороже
Нужно довести:
- spacing;
- typography;
- card rhythm;
- badge/chip language;
- alert language;
- toolbar composition;
- visual weight distribution.

## 18.2. Вторичные элементы должны стать тише
Признак зрелого premium UI — не крикливость, а уверенная иерархия.

## 18.3. Никаких случайных визуальных решений
Каждый крупный элемент должен выглядеть intentional.

---

# 19. Требования к top status/usage layer v2

## 19.1. System status strip должен быть легче
Он не должен:
- визуально доминировать;
- занимать слишком много пустой площади;
- выбиваться из product rhythm.

## 19.2. Он должен быть:
- компактным;
- полезным;
- премиально спокойным;
- явно вторичным относительно основного сценария.

---

# 20. Требования к локализации и копирайту v2

## 20.1. Интерфейс должен быть лексически консистентным
Нужно убрать хаотичное смешение:
- русского;
- английского;
- техно-лексики без системной причины.

## 20.2. Product vocabulary должен быть единым
Все:
- CTA;
- статусы;
- подписи;
- рекомендации;
- export labels;
- result messages

должны быть частью одной продуктовой речи.

## 20.3. Микрокопирайтинг должен отражать дорогой продукт
Тон:
- уверенный;
- короткий;
- понятный;
- без внутренней “технической болтовни”.

---

# 21. Accessibility и качество взаимодействия

## 21.1. Mobile touch
Всё важное должно быть:
- удобно нажимаемо;
- не слишком близко друг к другу;
- без конфликтов со scroll / overlay.

## 21.2. Desktop keyboard
Премиальный продукт не может быть keyboard-hostile в основном сценарии.

## 21.3. Mixed text direction
Иврит, русский, латиница и SBL должны выглядеть стабильно, предсказуемо и читаемо.

## 21.4. Motion discipline
Любые анимации или переходы должны усиливать ощущение зрелости, а не мешать восприятию.

---

# 22. Неприемлемые состояния после v2

`v2` нельзя считать успешным, если после него остаётся хотя бы часть следующих симптомов:

1. floating utility control визуально мешает;
2. stale-state не приводит к однозначной CTA hierarchy;
3. mobile first screen всё ещё длинный и шумный;
4. mobile table controls по-прежнему слишком низкоуровневые;
5. result surface не ощущается как единая сильная зона;
6. top status layer остаётся чрезмерно массивным;
7. интерфейс выглядит “хорошо”, но не выглядит “дорого и уверенно”.

## 24.1. Desktop DoD

2. Primary action hierarchy безошибочна.
3. Result surface цельная и сильная.
4. Usage/status layer не доминирует.
